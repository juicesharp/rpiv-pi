# Research Questions: Todo Tool Propagation into Parallel Subagents

## Discovery Summary

Discovery agents mapped the complete lifecycle: `todo.ts` stores state in module-level `let tasks: Task[]` and `let nextId = 1` (lines 60-61); `@tintinweb/pi-subagents` runs subagents in-process via `createAgentSession()` with a fresh `SessionManager.inMemory()` but re-uses the same Node.js module cache; child sessions fire `session.bindExtensions()` which triggers `session_start`, calling `reconstructTodoState(ctx)` that unconditionally resets `tasks = []; nextId = 1`. The integration scanner confirmed no IPC or event-based state synchronization exists between parent and child sessions. Key files read for depth: `todo.ts` (full), `index.ts` (full), `guidance.ts` (state section), `advisor.ts` (state section), `agent-runner.js` (lines 80-280), and the foundational migration research doc. Three extension modules (`todo`, `guidance`, `advisor`) all use the same module-level state pattern, making them all vulnerable to the same corruption class.

## Questions

1. Trace the exact corruption sequence when a subagent is spawned — from the `Agent` tool's `execute()` handler in `@tintinweb/pi-subagents/dist/index.js:630` through `runAgent()` in `agent-runner.js:99` which creates `SessionManager.inMemory()` at line 185, then `createAgentSession()` at line 188, then `session.bindExtensions()` at line 220 which fires `session_start` in the child — triggering `reconstructTodoState(ctx)` in `todo.ts:496` which unconditionally sets `tasks = []` and `nextId = 1`. Since Node.js caches ESM modules per process, the module-level `tasks`/`nextId` in `todo.ts:60-61` are shared across parent and child. The child's empty branch yields no entries, so the parent's entire todo state is wiped. This matters because every skill that spawns agents during active task tracking (research-questions, code-review) silently loses the parent's progress.

2. Explain how Node.js module caching creates the shared-state bug — when `DefaultResourceLoader.reload()` at `agent-runner.js:169` discovers extensions, and `session.bindExtensions()` triggers `import()` of `rpiv-core/index.ts`, does Node's ESM module registry return the cached module instance (with its existing closures over `tasks`/`nextId`), or does it create a fresh evaluation? `registerTodoTool(childPi)` at `todo.ts:578` registers a new tool on the child's `pi`, but the `execute` handler at line 587 captures `tasks` and `nextId` from the SAME module scope as the parent's handler. `getTodos()` at line 63 also reads the same shared `tasks`. This matters because any fix must either break the module-cache sharing or move state out of module scope.

3. Trace the background-agent concurrent mutation scenario — when `run_in_background: true` is used, the parent continues executing while the child runs concurrently in the same event loop. Both share `todo.ts`'s module-level `tasks`/`nextId`. Sequence: (a) parent creates tasks #1-5 via `execute()` at `todo.ts:587` which does `tasks = result.state.tasks`, (b) child's `session_start` fires `reconstructTodoState(ctx)` which does `tasks = []; nextId = 1` — wiping all 5 tasks, (c) child creates its own task #1, (d) parent calls `todo list` and sees only the child's task, (e) parent tries `todo update #3` — gets "task not found". Although JS is single-threaded (no interleaving within a single `execute` call), the `await` points in the agent lifecycle (between `bindExtensions()` and the first child tool call) allow interleaving. This matters because background agents are the primary use case for the research skill pipeline.

4. Analyze the recovery window between corruption and restoration — after a subagent's `session_start` wipes `tasks`/`nextId`, the parent's state remains corrupted until `reconstructTodoState(ctx)` runs again for the parent at `index.ts:107` (`session_compact`) or `index.ts:122` (`session_tree`). How long is this window in practice? For `research-questions`, the orchestrator creates 5-8 tasks, then spawns 4 parallel background agents — corruption happens at step (b). The parent only runs todo operations after all agents complete (Step 3 of the skill). `session_compact` fires only when the context window is nearly full. For typical research sessions, this means the todo state is corrupted for the ENTIRE duration of the parallel agent execution (potentially 5-10 minutes). The `todoOverlay` in `todo-overlay.ts` would show stale data because it only updates on `tool_execution_end` when `toolName === "todo"` in the parent session (line 128).

5. Evaluate whether `guidance.ts` and `advisor.ts` share the same vulnerability — `guidance.ts:59` has module-level `const injectedGuidance = new Set<string>()` cleared by `clearInjectionState()` at line 60, called from `session_start` at `index.ts:41`. `advisor.ts:114-115` has `let selectedAdvisor`/`let selectedAdvisorEffort` reset by `restoreAdvisorState()` at line 138, called from `session_start` at `index.ts:44`. When a child's `session_start` fires these, `clearInjectionState()` wipes the parent's guidance dedup set (causing re-injection of already-sent guidance messages to the parent), and `restoreAdvisorState()` may overwrite the parent's advisor selection with a file-based default. Does the child's guidance injection (when it reads files via `read` tool) cause duplicate messages in the parent's conversation? This matters because fixing only `todo.ts` leaves the same class of bug in two other tools.

6. Evaluate fix strategies for per-session state isolation — four approaches: (a) **WeakMap keyed on SessionManager**: `const stateMap = new WeakMap<SessionManager, {tasks, nextId}>()` — each session has its own `SessionManager` instance (parent uses file-backed, child uses `SessionManager.inMemory()`), so this naturally scopes state. `reconstructTodoState(ctx)` would read/write `stateMap.get(ctx.sessionManager)`. `getTodos()` needs a `ctx` parameter or a "current session" ref. (b) **Session ID from ExtensionAPI**: does `ExtensionAPI` expose any unique session identifier? If not, `WeakMap` keyed on `pi` itself (each `bindExtensions` creates a new `pi` instance) could work. (c) **Skip reconstruction in child sessions**: detect `ctx.sessionManager` is an in-memory instance (vs file-backed) and skip the wipe. But `execute()` still reads module-level state. (d) **Move state into `ExtensionAPI` context**: use `pi.setData("rpiv:todos", ...)` / `pi.getData("rpiv:todos")` if Pi's ExtensionAPI supports per-session data. Which approach is most robust and has the smallest blast radius on existing code?

7. Trace the `todoOverlay` interaction with shared-state corruption — the overlay at `todo-overlay.ts:44` reads `getTodos()` (module-level `tasks`) inside `renderWidget()` at line 114. When a child session wipes `tasks = []`, the overlay would render an empty list on next update. But `todoOverlay?.update()` at `index.ts:131` is only triggered by `tool_execution_end` when `toolName === "todo"` in the PARENT session — child session events don't bubble to the parent's overlay. So the overlay shows the pre-corruption state until the parent makes its next todo call, at which point it shows an empty list (or the child's tasks). Is this a secondary bug (overlay doesn't reflect real state) or just a symptom of the primary shared-state bug?

8. Map the blast radius across all rpiv-pi skills — which skills create todo tasks AND spawn subagents in the same session? `research-questions`: creates tasks at Step 2 then spawns agents — corruption window overlaps with active tracking. `research`: creates tasks at Step 3 then may spawn agents for each question. `design`: uses todos for vertical-slice tracking with parallel code generation. `code-review`: creates a review plan with todos then spawns parallel analysis agents. `implement-plan`: creates todos but spawns no agents (single orchestrator). `validate-plan`: reads todos but spawns no agents. For each affected skill, does the corruption manifest as: (a) parent loses all tasks, (b) parent sees child's tasks, (c) ID collisions between parent and child tasks?
