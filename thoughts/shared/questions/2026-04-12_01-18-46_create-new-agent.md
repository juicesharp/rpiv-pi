# Research Questions: Creating a New Agent in rpiv-pi

## Discovery Summary
Discovery agents scanned ~40 files across the rpiv-pi codebase, finding 9 bundled agent `.md` definitions in `agents/`, the auto-copy infrastructure in `extensions/rpiv-core/agents.ts` and `index.ts`, and 16 skill files that dispatch agents via prose instructions. Key files read for depth include 3 agent definition files (showing the frontmatter schema and system prompt structure), the `agents.ts` auto-copy utility, and the main `index.ts` extension entry point. The agent system relies on `@tintinweb/pi-subagents` (an external pi package) for runtime discovery, dispatch, and subprocess management — rpiv-pi only handles bundling and deployment of the `.md` files.

## Questions

1. **Trace how a new agent `.md` file in `agents/` is discovered, parsed, and made available at runtime.** Starting from `copyBundledAgents()` in `extensions/rpiv-core/agents.ts:36` which copies `agents/*.md` → `<cwd>/.pi/agents/`, through the `session_start` handler in `extensions/rpiv-core/index.ts:69-79` that triggers the copy, to `@tintinweb/pi-subagents`'s `reloadCustomAgents()` which scans `<cwd>/.pi/agents/*.md` and `~/.pi/agent/agents/*.md` on every `Agent` tool invocation, calling `parseFrontmatter()` from `@mariozechner/pi-coding-agent` to extract `name`, `description`, `tools`, `model`, `thinking`, `max_turns`, and the body as `systemPrompt`. The result is registered via `registerAgents()` which merges user agents (overriding same-name defaults). This matters because adding a new agent requires placing a correctly-formatted `.md` file in `agents/` and understanding this full pipeline.

2. **Explain the full lifecycle of an agent definition — from the frontmatter schema to the spawned pi process.** The `@tintinweb/pi-subagents` Agent tool (`src/index.ts:537-750`) receives parameters including `subagent_type`, `prompt`, `description`, `model?`, `thinking?`, `max_turns?`, `run_in_background?`, `resume?`, `isolated?`, `inherit_context?`, `isolation?`. It calls `resolveType(rawType)` for case-insensitive lookup against the unified registry (3 defaults + user agents). If unknown, it silently falls back to `"general-purpose"` (`src/index.ts:727-732`). The resolved agent's `tools` field determines which tools are available to the spawned `pi` subprocess via `--tools` flag filtering. The body of the `.md` file becomes the system prompt. This matters because understanding the fallback behavior and parameter passing is essential for debugging why a new agent might not be dispatched correctly.

3. **How does `@tintinweb/pi-subagents` resolve `subagent_type` against the agent registry, and what are the exact consequences of the silent `general-purpose` fallback for new agents?** The resolution chain is: `reloadCustomAgents()` → `loadCustomAgents(process.cwd())` → `loadFromDir()` for both global (`~/.pi/agent/agents/`) and project (`.pi/agents/`) dirs → `registerAgents()` clears registry, seeds 3 defaults (`general-purpose`, `Explore`, `Plan`), overlays user agents (project overrides global for same name) → `resolveType()` does case-insensitive Map lookup → returns `undefined` for unknown names → caller falls back to `"general-purpose"`. This matters because if a new agent's filename doesn't match expected patterns or frontmatter `name` is malformed, the agent silently degrades to generic behavior, and the 16 SKILL.md files that dispatch agents won't get the specialized behavior they expect.

4. **What is the complete contract between an agent `.md` file's YAML frontmatter fields and the runtime behavior of the dispatched agent?** The frontmatter supports: `name` (string, used for `subagent_type` matching), `description` (string, shown in tool description and `/agents` command), `tools` (CSV string like `grep, find, ls`, filtered against `BUILTIN_TOOL_NAMES`), `model` (optional, passed as `--model` flag to spawned pi), `thinking` (optional, passed as `--thinking` flag), `max_turns` (optional, limits agentic turns), `enabled` (optional, boolean), and the Markdown body becomes `systemPrompt`. Unknown tool names in the `tools` field are silently dropped by `getToolsForType()`. This matters because a new agent must specify exactly the right tool names and understand which fields are optional vs required for correct behavior.

5. **How do skills dispatch agents via prose instructions — trace from a SKILL.md file through to the actual Agent tool call?** Skills like `skills/research-questions/SKILL.md:40-49` use two patterns: (a) inline YAML with `subagent_type:` hints that the LLM translates into Agent tool parameters, and (b) prose instructions like "Use the Agent tool with subagent_type: `codebase-locator`" that tell the LLM to spawn specific agents. The LLM interprets these instructions and calls the `Agent` tool with the named `subagent_type`. This matters because a new agent must be referenced correctly in skill files, and the prose patterns must match the agent's registered name exactly — case-insensitive matching provides some leeway but the name must still resolve in the registry.

6. **What happens when a new agent needs tools beyond the built-in set (e.g., `web_search`, `web_fetch`) — how are tool names mapped to actual capabilities in the spawned subprocess?** The `web-search-researcher.md` agent specifies `tools: web_search, web_fetch, read, grep, find, ls` — the `@tintinweb/pi-subagents` filters tool names against `BUILTIN_TOOL_NAMES` which includes the standard pi tools. Extension-registered tools (like `web_search`, `web_fetch`) are available if the parent pi instance has them loaded. The `--tools` flag passed to the spawned `pi` subprocess restricts which tools are active. This matters because a new agent that needs non-standard tools must understand whether those tools are built-in, extension-provided, or unavailable, and whether the `tools` frontmatter field can reference them.
