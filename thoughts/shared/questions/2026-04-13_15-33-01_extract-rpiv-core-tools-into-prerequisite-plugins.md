# Research Questions: Assess extracting AskUserQuestion, Todos, and Advisor into separate prerequisite plugins

## Discovery Summary
Discovery found that the cleanest structural seams are already present at the module level: `extensions/rpiv-core/ask-user-question.ts`, `extensions/rpiv-core/todo.ts`, `extensions/rpiv-core/todo-overlay.ts`, and `extensions/rpiv-core/advisor.ts` are each imported into `extensions/rpiv-core/index.ts`, which acts as the current composition root for registration and lifecycle hooks. The highest-value packaging constraints sit outside the tool implementations themselves: `package.json` exposes only one local extension tree, `extensions/rpiv-core/package-checks.ts` plus `/rpiv-setup` in `index.ts` implement the existing sibling-plugin installation pattern, and `extensions/rpiv-core/permissions.ts` plus `extensions/rpiv-core/templates/pi-permissions.jsonc` centralize permission seeding. The 10 key files read for depth show three different extraction profiles: `ask_user_question` is almost stateless and UI-bound, `todo` couples tool logic with branch replay, `/todos`, and `TodoOverlay`, and `advisor` couples tool execution with config persistence, active-tool toggling, and model-registry access. Documentation and skill guidance currently assume these capabilities are baseline runtime features of `rpiv-core`, but most of that coupling appears to be contractual rather than code-level, which suggests that stable tool names and parameter shapes could preserve compatibility while ownership moves to prerequisite plugins.
