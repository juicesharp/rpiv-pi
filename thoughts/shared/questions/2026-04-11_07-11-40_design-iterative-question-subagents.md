# Research Questions: Parallel Question-Generation Subagents for `design-feature-iterative`

## Discovery Summary

Four parallel discovery agents mapped the ambiguity-identification machinery in both repo copies of `design-feature-iterative` (`rpiv-skillbased` and `rpiv-pi`), the decomposition + fan-out pattern of `research-questions` → `research`, the inbound/outbound contract between `design-feature-iterative` and its sole downstream consumer `write-plan`, and the prior-art thoughts/ docs (two chains directly relevant: the Option 1+4 fill-quality hardening design that introduced per-slice subagent dispatch, and the Holistic Self-Critique Loop design that added Step 3.5 via sub-numbering). The current Steps 3-5 ambiguity flow lives entirely in orchestrator prose at `rpiv-pi/skills/design-feature-iterative/SKILL.md:88-182`: Step 3 (Identify Ambiguities) triages 5 categories, Step 4 (Holistic Self-Critique) reviews the combined picture, Step 5 (Developer Checkpoint) asks grounded-questions-one-at-a-time via `ask_user_question`. None of the four sub-agents spawned at Step 2 (`codebase-pattern-finder`, `codebase-analyzer`, `integration-scanner`, `precedent-locator`) contains prompt scaffolds for surfacing design questions — that responsibility lives entirely in SKILL.md, not in agent files. Key files read for depth: `rpiv-pi/skills/design-feature-iterative/SKILL.md` (full 422 lines), `rpiv-skillbased/skills/design-feature-iterative/SKILL.md` (full), `rpiv-skillbased/skills/design-feature/SKILL.md` (full — sibling with no Step 4 self-critique), `rpiv-skillbased/skills/research-questions/SKILL.md` (full), `rpiv-skillbased/skills/research/SKILL.md` (full), `rpiv-skillbased/skills/write-plan/SKILL.md` (full), `rpiv-pi/extensions/rpiv-core/ask-user-question.ts` (full 116 lines), `rpiv-skillbased/thoughts/shared/designs/2026-04-09_12-38-44_option1-plus-option4-fill-quality-hardening.md:1-150`, `rpiv-skillbased/thoughts/shared/designs/2026-04-08_19-11-09_holistic-self-critique-loop.md:1-120`, and `rpiv-pi/thoughts/shared/questions/2026-04-11_10-40-21_todo-list-overlay-above-input.md` (format reference). Two load-bearing discoveries shape every question below: **(a)** Pi's `ask_user_question` tool schema at `rpiv-pi/extensions/rpiv-core/ask-user-question.ts:28-35` accepts exactly ONE question per call (single `question: Type.String`, no array), making the "batch 2-4 in a single call" instruction at `rpiv-pi/skills/design-feature-iterative/SKILL.md:153` factually wrong — a carryover from Claude Code's native array-taking `AskUserQuestion`; **(b)** `write-plan/SKILL.md:26` extracts exactly six sections from the design artifact (Architecture, File Map, Ordering Constraints, Verification Notes, Performance Considerations, Scope) and STOPS at `:29`, `:253`, `:301` if unresolved questions remain — these six sections define the "question-worthy" surface, while the Design History slice-log at `design-feature-iterative/SKILL.md:240` is explicitly ignored by write-plan. Prior art: no document in either repo currently discusses parallel question generation; the closest is the Option 1+4 design which parallelized Step 7 code *fill* via per-slice subagents but explicitly deferred named-agent definitions ("agent dispatch patterns cascade — 3 pattern changes in 2 days at commits 441ef08→b75f2aa").

## Questions

1. Trace how the ambiguity surface of `design-feature-iterative` can be decomposed into parallelizable areas analogous to `research-questions` spawning one `codebase-locator` per research-query area. The current triage at `rpiv-pi/skills/design-feature-iterative/SKILL.md:88-102` enumerates 5 ambiguity types (Pattern conflict / Missing pattern / Scope boundary / Integration choice / Novel approach); the research-questions fan-out at `rpiv-pi/skills/research-questions/SKILL.md:42-49` spawns one locator per decomposed area plus `integration-scanner` and `thoughts-locator`. The question is whether architectural dimensions — data model, API surface, integration wiring, scope, verification, performance, migration — form stabler parallelization axes than the 5 ambiguity types, and whether those dimensions map cleanly onto the 6 write-plan-consumed sections (Architecture, File Map, Ordering Constraints, Verification Notes, Performance Considerations, Scope) listed at `rpiv-skillbased/skills/write-plan/SKILL.md:26`. This matters because the decomposition axis determines each sub-agent's context window and what coverage guarantee the orchestrator must make after fan-in, exactly mirroring research-questions' "every key file must appear in at least one question" rule at `rpiv-skillbased/skills/research-questions/SKILL.md:100`.

2. Trace what a "design-question-generator" sub-agent actually looks like as a file, by walking the existing agent-definition pattern — each is a plain Markdown file with YAML frontmatter declaring `tools:` (e.g. `rpiv-skillbased/agents/codebase-locator.md` with `tools: Grep, Glob, LS` and Pi's lowercase `rpiv-pi/agents/codebase-locator.md` with `tools: grep, find, ls`) and a body with strategy + output format, with no TS registration — and by contrasting with the Option 1+4 precedent at `rpiv-skillbased/thoughts/shared/designs/2026-04-09_12-38-44_option1-plus-option4-fill-quality-hardening.md:131-150` which explicitly chose a general-purpose agent with 7 assembled prompt blocks (artifact path, slice spec, pattern templates, prior-slices summary, decisions & constraints, behavioral priority rule, self-verify instructions) over a named agent, citing cascading pattern changes as the rationale. The new agent would need analogous blocks: research artifact path, assigned architectural dimension, pattern templates from Step 2 at `design-feature-iterative/SKILL.md:60-76`, the "thoughts/ docs are NOT questions" rule from `research-questions/SKILL.md:96`, the "pre-validate options before presenting" rule from `design-feature-iterative/SKILL.md:102`, and a dense-paragraph output schema mirroring the question artifacts in `thoughts/shared/questions/`. Decide whether v1 uses general-purpose-with-assembled-prompt (the Option 1+4 precedent) or ships new `.md` files into `rpiv-skillbased/agents/`, `rpiv-pi/agents/`, AND `rpiv-pi/.pi/agents/` via the auto-copy at `rpiv-pi/extensions/rpiv-core/agents.ts:36-62`.

3. Resolve the Pi `ask_user_question` single-question schema vs the false "batching" claim currently in the SKILL.md. Trace the exact parameter schema at `rpiv-pi/extensions/rpiv-core/ask-user-question.ts:28-35` — `parameters: Type.Object({ question: Type.String(...), header: Type.Optional(...), options: Type.Array(OptionSchema), multiSelect: Type.Optional(...) })` accepting **one** question per call — and contrast with `rpiv-pi/skills/design-feature-iterative/SKILL.md:153` which still instructs "you MAY batch 2-4 in a single `ask_user_question` call" (a carryover from Claude Code's native `AskUserQuestion` which genuinely takes a `questions` array, visible in skill YAML examples at `rpiv-skillbased/skills/design-feature-iterative/SKILL.md:150-159`). If N parallel sub-agents each surface M questions, the Pi orchestrator must make N×M sequential `ask_user_question` tool calls, each blocking on the previous and each amplifying the "ask one at a time" rule at `design-feature-iterative/SKILL.md:140`. Decide: (a) accept sequential-only with a hard cap (e.g. 5 questions per design run), (b) add a new array-taking multi-question tool to `rpiv-pi/extensions/rpiv-core/` to parallel Claude Code's native one (register it in `index.ts:23,28` alongside `registerAskUserQuestionTool`), or (c) force sub-agents to self-prioritize so only the top 1-3 architectural questions survive fan-in — and whether the line-153 bug gets fixed as part of this work or separately.

6. Trace how fan-in rules interact with the existing Holistic Self-Critique at `rpiv-pi/skills/design-feature-iterative/SKILL.md:104-122`, which today performs orchestrator-side review for cross-cutting concerns, pattern coherence, and ambiguity completeness, and can reclassify Step 3 ambiguities as simple/genuine before they reach Step 5. When N parallel sub-agents each produce questions without seeing each other's output, two of them may surface the same ambiguity from different angles (e.g. a "data-model" sub-agent and a "migration" sub-agent both flagging a schema-version field), and one may surface a question whose options are already invalidated by constraints found by another. Decide who does the merge, dedupe, and pre-validation (per `:102` "check every option against research constraints"): (a) the orchestrator merges sub-agent outputs and THEN runs the existing Step 4 critique, which now reviews the combined picture, preserving the "fix in-place or add to checkpoint queue" remediation at `:119-122`; (b) a meta sub-agent runs after the N parallel ones and performs the critique (closer to research-questions' Step 4 synthesize at `rpiv-skillbased/skills/research-questions/SKILL.md:78-100`); (c) each parallel sub-agent runs its own dimension-local critique before returning. Option (a) is the smallest delta to existing prose, but amplifies the orchestrator's context load exactly when the goal of parallelization is to reduce it — the same tension the Option 1+4 design resolved by moving self-verify **inside** the fill subagent at `rpiv-skillbased/thoughts/shared/designs/2026-04-09_12-38-44_option1-plus-option4-fill-quality-hardening.md:79-82`.

The two are complementary, not exclusive — decide whether v1 ships one or both, and if per-slice, whether to reuse the Option 1+4 prior-slices-summary block (the compressed type/interface/signature context assembled at lines 140-147 of that prior design) to give each per-slice question sub-agent the same cross-slice context that the fill sub-agent already receives.
