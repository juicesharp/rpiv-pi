# Research Questions: Advisor Settings Persistence Between Sessions

## Discovery Summary
Discovery agents mapped the complete advisor implementation in `extensions/rpiv-core/advisor.ts` (525 lines, single-file) where two module-level `let` variables — `selectedAdvisor: Model<Api> | undefined` and `selectedAdvisorEffort: ThinkingLevel | undefined` — store the user's chosen advisor model and reasoning effort level. These are explicitly in-memory only and reset each session (documented at line 13). The existing codebase has two distinct persistence patterns to draw from: (1) file-based config persistence in `extensions/web-tools/index.ts` with `loadConfig()`/`saveConfig()` writing JSON to `~/.config/rpiv-pi/web-tools.json`, and (2) session-entry-based state reconstruction in `extensions/rpiv-core/todo.ts` via `reconstructTodoState()` walking `ctx.sessionManager.getBranch()`. The `session_start` handler in `extensions/rpiv-core/index.ts` already reconstructs todo state on startup. The `/advisor` command at `advisor.ts:323-525` uses `ctx.ui.custom()` with a `SelectList` to pick models from `ctx.modelRegistry.getAvailable()`. Key files read for depth: `advisor.ts`, `index.ts`, `web-tools/index.ts`, `package-checks.ts`, plus the agent-based advisor feasibility questions and the original advisor design doc that explicitly rejected cross-session persistence as out-of-scope.

## Questions

1. Trace the file-based config persistence pattern used by `extensions/web-tools/index.ts` and evaluate its applicability to advisor settings. The web-tools extension defines a `WebToolsConfig` interface (line 32) with an `apiKey?: string` field, a `CONFIG_PATH` constant pointing to `~/.config/rpiv-pi/web-tools.json` (line 36), a `loadConfig()` function (line 41) that reads JSON with `readFileSync` and returns `{}` on missing/error, and a `saveConfig()` function (line 50) that writes JSON with `writeFileSync`, creates parent directories with `mkdirSync`, and sets `chmod 0o600`. The `/web-search-config` command handler (lines 459–492) calls `saveConfig()` after user input. For advisor persistence, what would the config schema look like — storing `advisorModel: string` (as `"provider:id"`) and `advisorEffort: string` (as a ThinkingLevel literal) — and should it reuse `~/.config/rpiv-pi/` with a new file (e.g., `advisor.json`) or embed into an existing config? This matters because the web-tools pattern is the only example of user-settings persistence across extensions in this codebase.

2. Trace the session-start lifecycle in `extensions/rpiv-core/index.ts` to identify where advisor state restoration would hook in. The `session_start` handler (lines 40–94) runs in order: `clearInjectionState()` → `reconstructTodoState(ctx)` → `TodoOverlay` setup → `active_agent` entry seeding → `thoughts/` directory scaffolding → bundled agent copying → permissions file seeding → subagent install warning. After this, `registerAdvisorBeforeAgentStart` at `advisor.ts:283-290` runs every turn and strips `"advisor"` from active tools if `getAdvisorModel()` returns `undefined`. To restore persisted advisor settings, a restoration function would need to: (a) load the config file, (b) resolve the stored `"provider:id"` string back to a `Model<Api>` object via `ctx.modelRegistry.getAvailable()`, (c) call `setAdvisorModel()` and `setAdvisorEffort()`, and (d) add `"advisor"` to active tools via `pi.setActiveTools()`. Where exactly in the `session_start` sequence should this restoration happen — after `reconstructTodoState(ctx)` but before the todo overlay setup — and what happens if the stored model is no longer in the registry?

3. Trace the model resolution pathway from a persisted `"provider:id"` string back to a live `Model<Api>` object. The `/advisor` command at `advisor.ts:332` gets available models via `ctx.modelRegistry.getAvailable()` which returns `Model<Api>[]`. The command uses a `modelKey()` helper (line 319) that produces `"${m.provider}:${m.id}"` strings for matching. When restoring from config, the code would need to call `ctx.modelRegistry.getAvailable()` during `session_start`, find the model matching the stored key, and pass it to `setAdvisorModel()`. However, `ExtensionContext` with its `modelRegistry` is only available inside event handlers — the `session_start` callback receives `(event, ctx)` at `index.ts:40`. Investigate: does `ctx.modelRegistry.getAvailable()` work correctly at `session_start` time (i.e., are providers already initialized), and what is the graceful degradation path when the stored model key no longer matches any available model (provider removed, model retired, API key revoked)?

4. Trace the `reconstructTodoState()` pattern in `extensions/rpiv-core/todo.ts:496-575` and compare its trade-offs against file-based persistence for advisor settings. The todo tool persists state by encoding task arrays in `AgentToolResult.details` envelopes that get serialized into the session JSONL transcript. On session start and after compaction, `reconstructTodoState(ctx)` walks `ctx.sessionManager.getBranch()` looking for the last `todo` tool result, parses the `TaskDetails` from it, and restores the module-level `tasks[]` and `nextId` state. This approach survives compaction (pre-compaction entries are preserved) but does NOT survive new sessions (different JSONL file). For advisor persistence, this session-entry approach would be unsuitable because the goal is cross-session persistence — a new session starts with a fresh JSONL transcript. This matters because it establishes that file-based config (web-tools pattern) is the correct approach for cross-session advisor settings, not the todo's session-entry pattern.

5. Trace what happens during Pi's `session_compact` event at `extensions/rpiv-core/index.ts:104-108` and evaluate whether advisor state needs explicit handling during compaction. The current handler calls `clearInjectionState()` and `reconstructTodoState(ctx)` but does NOT touch advisor state. The advisor's in-memory `selectedAdvisor` and `selectedAdvisorEffort` variables survive compaction unchanged because they are module-level state, not derived from session entries. With file-based persistence added, compaction would have no impact on the config file — it persists on disk independently. However, investigate whether the `before_agent_start` handler at `advisor.ts:283-290` could inadvertently strip the advisor tool during the first turn after compaction if state restoration happens in `session_start` but not in `session_compact`. Since the advisor state is module-level (surviving compaction without reconstruction), this should be a non-issue — but verify that `session_compact` does NOT reload extensions or reset module state.

6. Trace the interaction between persisted advisor settings and Pi's own global model configuration at `~/.pi/agent/settings.json`. The `readInstalledPackages()` function in `extensions/rpiv-core/package-checks.ts:21` reads `~/.pi/agent/settings.json` for the `packages[]` array, proving that extensions can safely read Pi's global settings. Pi's settings also store `defaultProvider`, `defaultModel`, and `defaultThinkingLevel`. Should the advisor config file reference Pi's global model definitions (e.g., storing just `defaultModel` to mean "use whatever Pi's default model is") or should it store explicit `"provider:id"` keys? Consider the scenario where a user changes their default model in Pi's settings — should the advisor automatically follow, or should the advisor selection be independently pinned? This matters because the advisor is conceptually "a stronger model than the executor," so tying it to Pi's default might not make semantic sense (the default might already be the strongest model).

7. Trace the `/advisor` command's model selection UI at `extensions/rpiv-core/advisor.ts:323-525` and determine how to indicate a persisted selection on the selector screen. The current UI builds `SelectItem[]` with a `check` suffix appended to the current model's label (line 396: `` `${m.name}  (${m.provider})${check}` ``). When a persisted advisor is restored on session start, `getAdvisorModel()` would return the restored model, so the `✓` marker would automatically appear on the correct item. However, the effort picker at lines 430-495 does NOT show the current selection — it always defaults to `high` via `selectList.setSelectedIndex(baseLevels.indexOf("high") + 1)`. If effort is persisted and restored, the effort picker should pre-select the stored effort level instead. Additionally, should the `/advisor` command display a hint like "(restored from previous session)" to inform the user that the selection was auto-loaded?

8. Trace the security and privacy considerations of persisting advisor model selection to disk. The web-tools config at `~/.config/rpiv-pi/web-tools.json` stores an API key and sets `chmod 0o600` (owner read/write only). The advisor config would store a model identifier string (e.g., `"anthropic:claude-opus-4-6"`) and an effort level string (e.g., `"high"`) — no API keys or sensitive credentials. Evaluate: does the model identifier alone constitute sensitive information (it reveals which AI service the user subscribes to), should the config file also use `chmod 0o600`, and are there any implications for multi-user systems where `~/.config` might be on shared storage? Additionally, consider whether the config file should be per-project (stored in `.rpiv/` under the project root) or global (stored in `~/.config/rpiv-pi/`), given that different projects might benefit from different advisor models (e.g., a Python project might prefer a code-specialized model while a documentation project might prefer a generalist).
