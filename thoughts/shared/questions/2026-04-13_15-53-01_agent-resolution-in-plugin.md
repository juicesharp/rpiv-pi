# Research Questions: How agents are resolved in the current plugin

## Discovery Summary
Agent resolution in rpiv-pi spans two packages: the in-repo `extensions/rpiv-core` extension (bundled-agent auto-copy, package-presence checks, active-agent seeding) and the sibling `@tintinweb/pi-subagents` package (custom-agent loading, registry merging with built-in defaults, Agent-tool dispatch). Discovery located ~20 files: the 9 bundled `.md` profiles in `agents/`, three rpiv-core utility modules (`agents.ts`, `package-checks.ts`, `index.ts`), and five pi-subagents modules installed at `/usr/local/lib/node_modules/@tintinweb/pi-subagents/src/` (`custom-agents.ts`, `agent-types.ts`, `agent-manager.ts`, `agent-runner.ts`, `index.ts`). The two halves communicate only through the filesystem: rpiv-core copies profiles into `<cwd>/.pi/agents/` at `session_start`; pi-subagents scans that directory on init and on every `Agent` tool invocation to build a runtime map keyed by filename stem. Key files read for depth: `extensions/rpiv-core/agents.ts`, `extensions/rpiv-core/index.ts`, `extensions/rpiv-core/package-checks.ts`, `agents/codebase-locator.md`, plus `custom-agents.ts`, `agent-types.ts`, and the `Agent` tool's `execute()` handler in pi-subagents' `index.ts`.

## Questions

1. Trace the **bundled-agent auto-copy path** at session start — from `pi.on("session_start", async (_event, ctx) => { ... })` in `extensions/rpiv-core/index.ts:41`, through `copyBundledAgents(ctx.cwd, false)` in `extensions/rpiv-core/agents.ts:36`, and its `PACKAGE_ROOT` resolution via `import.meta.url + fileURLToPath` at `agents.ts:19-23`. Cover: how it walks `dirname()×3` from `extensions/rpiv-core/agents.ts` to the package root, how `BUNDLED_AGENTS_DIR = join(PACKAGE_ROOT, "agents")` is derived, why skip-if-exists is the default (protecting local edits to `<cwd>/.pi/agents/*.md`), and how the `/rpiv-update-agents` command at `index.ts:162-176` forces `overwrite=true` to refresh bundled copies. Explain why bundled files land in `<cwd>/.pi/agents/` rather than `~/.pi/agent/agents/`, and what happens when `BUNDLED_AGENTS_DIR` does not exist.

2. Trace the **two-tier discovery + priority override** performed by pi-subagents — from `loadCustomAgents(cwd)` in `/usr/local/lib/node_modules/@tintinweb/pi-subagents/src/custom-agents.ts:21`, which loads `~/.pi/agent/agents/` first with `source="global"` (lower priority) then `<cwd>/.pi/agents/` with `source="project"` (higher priority, overwrites by key) via `loadFromDir()` at `custom-agents.ts:32`, and into `registerAgents(userAgents)` at `agent-types.ts:44` which seeds the map with `DEFAULT_AGENTS` (`general-purpose`, `Explore`, `Plan` from `default-agents.ts`) then overlays user agents so user names override defaults. Explain why rpiv-pi's auto-copy into `.pi/agents/` guarantees its agents win over any global agents and over the built-in defaults when names collide, and why the registry lives in a module-level `Map<string, AgentConfig>` at `agent-types.ts:37`.

3. Trace the **frontmatter → AgentConfig parsing pipeline** — from `loadFromDir()` calling `parseFrontmatter<Record<string,unknown>>(content)` at `custom-agents.ts:52` and then populating every `AgentConfig` field through the per-field extractors `str()`, `csvList()`, `parseCsvField()`, `nonNegativeInt()`, `inheritField()`, and `parseMemory()` defined at `custom-agents.ts:78-137`. Apply this to rpiv-pi's actual frontmatter shape (`name`, `description`, `tools`) as documented in `.rpiv/guidance/agents/architecture.md` and shown in `agents/codebase-locator.md:1-5`. Cover: how `tools: grep, find, ls` becomes `builtinToolNames` via `csvList(fm.tools, BUILTIN_TOOL_NAMES)` against the registry at `agent-types.ts:34`, the `"none"` sentinel behavior, the `enabled !== false` default, and the load-bearing detail that `name` in the map is `basename(file, ".md")` — so the filename stem (not the frontmatter `name` field) is authoritative.

4. Trace the **runtime Agent-tool invocation and type resolution** — from the `Agent` tool's `execute()` in `/usr/local/lib/node_modules/@tintinweb/pi-subagents/src/index.ts:723`, which first calls `reloadCustomAgents()` (re-reads `.pi/agents/*.md` each call so newly-added profiles are picked up without restart, defined at `index.ts:248`), then `resolveType(rawType)` in `agent-types.ts:69` for case-insensitive lookup, and the silent fallback `subagentType = resolved ?? "general-purpose"` with a `fellBack` flag at `index.ts:732-733`. Follow on into `getAgentConfig(subagentType)` → `resolveAgentInvocationConfig(customConfig, params)` at `index.ts:738-740`, and finally into `manager.spawn(pi, ctx, subagentType, params.prompt, { ... })` in `agent-manager.ts:85` that dispatches to `runAgent()` in `agent-runner.ts`. Explain why resolution happens per-call (not cached across the session) and how an unknown `subagent_type` fails open to `general-purpose`.

5. Document the **tool-set-determines-tier contract** that shapes what each resolved agent can do — from `TOOL_FACTORIES` in `agent-types.ts:23-31` listing `read`, `bash`, `edit`, `write`, `grep`, `find`, `ls` and the derived `BUILTIN_TOOL_NAMES` exported at `agent-types.ts:34`, through the per-agent `builtinToolNames` parsed from the frontmatter `tools:` list, and how this intersects with the architectural boundary documented in `.rpiv/guidance/agents/architecture.md` ("locators have no `read`"; only `precedent-locator` gets `bash`; only `web-search-researcher` gets web tools — those are extension-registered, not builtin). Explain why no agent in `.pi/agents/` can dispatch another agent: the `Agent` / `get_subagent_result` / `steer_subagent` names are listed in `EXCLUDED_TOOL_NAMES` at `agent-runner.ts:25` and stripped from inherited tools before session creation.

6. Trace the **sibling-package dependency and bootstrapping flow** that must succeed for agent resolution to work — from the `hasPiSubagentsInstalled()` / `hasPiPermissionSystemInstalled()` checks in `extensions/rpiv-core/package-checks.ts:33-39` (both read `~/.pi/agent/settings.json` `packages[]` via `readInstalledPackages()`), through the session-start warning at `extensions/rpiv-core/index.ts:98-104` when pi-subagents is missing, and the `pi.appendEntry("active_agent", { name: "general-purpose" })` seed at `index.ts:64-66` that prevents pi-permission-system's input handler from blocking a first-message `/skill:<name>` with "active agent context is unavailable". Cover how `/rpiv-setup` (`index.ts:179-262`) detects missing packages and calls `pi install <pkg>` for each, and what silent fallback behavior skills exhibit when pi-subagents is absent entirely (no `Agent` tool registered).

7. Trace the **model resolution override chain** for a resolved agent — from `resolvedConfig.modelInput` in the `Agent` tool's `execute()` at `pi-subagents/src/index.ts:743-752`, through `resolveModel(input, ctx.modelRegistry)` and the internal `resolveDefaultModel()` in `agent-runner.ts:53-78` (priority: `configModel` split on `/` → `registry.find(provider, modelId)` → `registry.getAvailable()` membership check → fall through to `parentModel`). Show how a `model:` line in an rpiv-pi agent's frontmatter (e.g., `Explore`'s default `anthropic/claude-haiku-4-5-20251001` in `default-agents.ts`) flows through `str(fm.model)` in `custom-agents.ts:63` into `AgentConfig.model`, out through `resolveAgentInvocationConfig`, and why the "config-specified → silent fallback to parent" branch at `pi-subagents/src/index.ts:748` behaves differently from the "tool-call params specified → return error text" branch — so tool-call params only fill gaps the agent config left unset.
