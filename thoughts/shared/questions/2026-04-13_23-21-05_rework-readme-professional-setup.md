# Research Questions: Rework README to professional standard with clean setup

## Discovery Summary

Seven parallel discovery agents mapped the full rpiv-pi surface area: 17 skill SKILL.md files, 9 agent definitions, the rpiv-core extension registration surface (hooks, commands, tools), all setup/config files, the extracted web-tools sibling, the scripts/ directory, and 10 prior thoughts docs about README rework and plugin extraction. The main orchestrator reads 11 key files for depth: `README.md`, `package.json`, `extensions/rpiv-core/index.ts` (243 lines — session lifecycle, `/rpiv-setup`, `/rpiv-update-agents`), `package-checks.ts` (5 sibling detection functions reading `~/.pi/agent/settings.json`), `guidance.ts` (per-depth guidance resolution and injection), `agents.ts` (bundled-agent auto-copy), `.rpiv/guidance/skills/architecture.md` (skill pipeline and SKILL.md conventions), `.rpiv/guidance/agents/architecture.md` (agent capability tiers), the prior research-questions artifact on this topic, the plugin-extraction design doc's README spec section, and the `rpiv-advisor/README.md` sibling convention reference. A prior research-questions artifact (`questions/2026-04-13_21-57-35_rework-readme-professional-setup.md`) exists but has no downstream research/design/plan artifacts. The setup flow is fully implemented; the README rework is a documentation-only change.

## Questions

1. Trace every pre-existing dependency a new user must have before `pi install rpiv-pi` can succeed, from the `peerDependencies` block in `package.json:11-21` (which names `@mariozechner/pi-ai`, `@mariozechner/pi-coding-agent`, `@mariozechner/pi-tui`, `@sinclair/typebox`, and the five sibling plugins), through the `readInstalledPackages()` reader in `extensions/rpiv-core/package-checks.ts:21-31` (which parses `~/.pi/agent/settings.json`), to the implicit requirements this reader assumes — a Pi runtime already installed with a populated `settings.json`, Node.js, and git for the `before_agent_start` hook at `index.ts:104-121`. Classify every dependency as *user-installs-first* (Pi CLI, Node, git), *bundled-with-Pi-runtime* (pi-ai, pi-coding-agent, pi-tui, typebox), or *rpiv-family-installed-by-setup* (the five siblings). The README's "Prerequisites" section must list only the first category with concrete install commands; the others belong under separate sections. This matters because listing peerDependencies as prerequisites creates a confusing onboarding experience for brand-new users who don't know which ones Pi already provides.

2. Trace the end-to-end first-run flow a new user experiences, from `pi install rpiv-pi` through the first productive session, naming every observable user-facing event: Pi loading the extension via the default export at `index.ts:32`, the `session_start` handler at `index.ts:34-85` that scaffolds `thoughts/shared/{research,questions,designs,plans,handoffs}` directories (`index.ts:50-59`) and calls `copyBundledAgents()` from `agents.ts:36-62` (writing nine `.md` files to `<cwd>/.pi/agents/`), the aggregated missing-plugin warning at `index.ts:70-84` using the five `has*Installed()` checks from `package-checks.ts:33-55`, the user running `/rpiv-setup` (handler at `index.ts:141-242`) with its confirm dialog at `index.ts:189-196` and sequential `pi.exec("pi", ["install", pkg])` loop at `index.ts:206-224`, the "Restart your Pi session" footer at `index.ts:238`, and `/web-search-config` for the Brave API key. This exact ordering dictates the numbered-steps block the README must use — deviating from the code's real sequence will produce onboarding bugs where users try steps in the wrong order.

3. Resolve the skill-invocation prefix by examining how Pi's runtime discovers and exposes skills declared via `"skills": ["./skills"]` in `package.json:9`. The current README uses `/skill:<name>` at lines 109-114 in its "Typical workflow" block, while `rpiv-skillbased/README.md` uses `/rpiv-next:<name>` throughout. The skills architecture doc at `.rpiv/guidance/skills/architecture.md` confirms each `SKILL.md` frontmatter `name` field maps to the invocation command, but does not specify the prefix. Determine whether the prefix is owned by Pi's runtime (making `/skill:` authoritative), by the package name, or by a skill-level configuration. The README's workflow examples must use the exact prefix a Pi user would type; mixing conventions or documenting an incorrect prefix breaks the first interactive step after installation.

4. Map how the README should present the 17 skills, 9 agents, and their pipeline relationships, based on the skill pipeline documented in `.rpiv/guidance/skills/architecture.md` (research-questions → research → design → write-plan → implement-plan → validate-plan) and the agent capability tiers documented in `.rpiv/guidance/agents/architecture.md` (locators: grep/find/ls only; analyzers: +read; pattern-finders: +read; integration-scanner: connection mapping; precedent-locator: +bash/git; web-search-researcher: +web tools). The current README lists all 17 skills in a flat table and 9 agents in another flat table without showing the pipeline order or which skills consume which agents. Decide whether the README should show pipeline dependencies graphically, use section groupings (Research, Design, Implementation, Testing, Annotation), or keep the flat table but add a "Pipeline" section showing the chain. This matters because a new user needs to understand that skills form an ordered pipeline, not an à-la-carte menu.

5. Audit the four sibling READMEs (`rpiv-advisor/README.md`, `rpiv-ask-user-question/README.md`, `rpiv-todo/README.md`, `rpiv-web-tools/README.md`) for shared conventions — one-paragraph lede under the `#` title, a `docs/*.jpg` screenshot, a 4-space-indented `pi install npm:@juicesharp/…` prose block, a "Then restart your Pi session" trailer, and a `## License\n\nMIT` footer — and decide item-by-item which the parent `rpiv-pi/README.md` should adopt. Specifically resolve: (a) whether to replace the bare `Version: 0.4.0` line at `README.md:5` with a descriptive lede paragraph, (b) whether to add a `## License` section (currently absent), (c) whether to add a screenshot or deliberately omit one since rpiv-pi is an orchestrator with no single UI element, (d) whether to keep fenced `bash` code blocks for the multi-step install flow or switch to the sibling convention of 4-space-indented prose. For each decision, state whether aligning helps brand-new-user onboarding or whether the parent's orchestrator role justifies divergence.

6. Decide the granularity at which the README should document what happens invisibly on every session start, tracing through the five operations in `index.ts:34-85`: `clearInjectionState()` + `injectRootGuidance()` at lines 35-36 (reading `.rpiv/guidance/architecture.md`), the `active_agent` seed workaround at lines 45-47 (conditional on `hasPiPermissionSystemInstalled()` from `package-checks.ts:37-39`), the `mkdirSync` directory scaffolding at lines 50-59, the idempotent `copyBundledAgents(cwd, false)` at line 62 (from `agents.ts:36-62`), and the `tool_call` guidance-injection hook at lines 99-101 (implemented in `guidance.ts:162-192`). The README must either surface these as a concise "What happens on session start" section or deliberately omit internal details. This decision affects whether users understand why `.pi/agents/` appears in their project, why `thoughts/shared/` is scaffolded, and why `/rpiv-update-agents` exists — all of which are currently undocumented behaviors that surprise new users.

7. Audit the existing `## Migration from 0.3.x` section at `README.md:116-125` against the design doc acceptance criterion in `thoughts/shared/designs/2026-04-13_17-00-00_extract-rpiv-plugins.md` (which requires the README to reference all four `@juicesharp/rpiv-*` plugin names and explain the silent config-path cutover from `~/.config/rpiv-pi/` to plugin-owned paths). Map every concrete failure mode the code emits: `"/rpiv-setup requires interactive mode"` in headless mode (`index.ts:144-147`), per-package failure `"✗ Failed: ${pkg}: ${error}"` (`index.ts:231-234`), the aggregated missing-sibling warning (`index.ts:79-82`), and the "4 concurrent background agents" throttle note currently buried at `README.md:131`. Decide whether a dedicated "Troubleshooting" section is warranted and list every actionable failure the README should pre-empt with exact remediation text. This matters because the current README has no troubleshooting guidance, and every failure mode listed here is one a new user will encounter during first-run.

8. Determine the optimal professional README structure for a Pi CLI plugin package by examining industry-standard conventions for developer-tool OSS projects: section ordering (title → badges → one-paragraph description → features → prerequisites → quick-start/install → usage → configuration → architecture overview → contributing → license), whether to add shields.io badges for version/license/platform, whether a table-of-contents is warranted for the current ~130-line README or only if it grows beyond ~200 lines, and whether the `## Notes` section at `README.md:129-132` should be promoted into the main body or remain as a footnote section. Compare the current structure (title → version → requirements → installation → what's included → workflow → migration → notes) against what a professional OSS README typically includes. This matters because the user explicitly requested the README "look highly professional and follow best industry standards."
