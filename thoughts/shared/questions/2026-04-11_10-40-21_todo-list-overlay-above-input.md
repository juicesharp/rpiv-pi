# Research Questions: Todo List Overlay Above User Input

## Discovery Summary

Six parallel discovery agents mapped the interactive-mode render pipeline, the subagent reference implementation, the todo tool integration, existing scroll patterns, the event/session wiring, and prior design decisions from `thoughts/`. The canonical mount point is `InteractiveMode.setExtensionWidget()` at `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/dist/modes/interactive/interactive-mode.js:1112`, reached via `ctx.ui.setWidget(key, factory, { placement: "aboveEditor" })` — the exact mechanism `@tintinweb/pi-subagents`' `AgentWidget` uses at `/usr/local/lib/node_modules/@tintinweb/pi-subagents/dist/ui/agent-widget.js:380`. Widgets are stored in `extensionWidgetsAbove: Map<string, Component>` (interactive-mode.js:128-131) and flushed into `widgetContainerAbove` (mounted at line 370 between `statusContainer` and `editorContainer`) through `renderWidgetContainer()` at lines 1194-1208, which injects a leading `Spacer(1)` and concatenates components with no height cap for the factory form (the 10-line `MAX_WIDGET_LINES` cap at line 1183 applies only to `string[]` content). Key files read for depth: `extensions/rpiv-core/todo.ts` (full), `extensions/rpiv-core/index.ts` (full), `pi-subagents/dist/ui/agent-widget.js` (full), `pi-subagents/dist/ui/conversation-viewer.js` (full), `pi-coding-agent/dist/modes/interactive/interactive-mode.js:1100-1330`, `pi-tui/dist/tui.js:125-320,500-580`, and `pi-coding-agent/dist/core/extensions/types.d.ts:55-164`. Two prior thoughts docs — `thoughts/shared/research/2026-04-10_21-53-11_todo-tool-cc-parity.md` and `thoughts/shared/designs/2026-04-10_22-34-39_todo-tool-cc-parity.md` — previously rejected an overlay citing "pi-tui overlays hard-truncate at maxHeight (`tui.js:565-567`) with no built-in scrolling"; this research deliberately revisits that decision now that the user has accepted building scroll state inside the widget itself. Critical gaps surfaced: (a) the `tool_execution_end` event fires BEFORE the toolResult is appended to `sessionManager.getBranch()` (see `agent-session.js:265-303`), so an overlay must either read `event.result.details` directly or subscribe to `message_end`; (b) `widgetContainerAbove` children never receive `handleInput` because the editor always owns focus, so scroll keys must come through `TUI.addInputListener` (reachable via `ctx.ui.onTerminalInput`) or a `nonCapturing: true` overlay; (c) `OverlayOptions.maxHeight` at `tui.js:527` truncates and does not scroll — the scroll-and-slice must live inside the widget's own `render(width)`, mirroring `ConversationViewer.render()`.
