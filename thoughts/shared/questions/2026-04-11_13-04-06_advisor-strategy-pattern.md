# Research Questions: Advisor Strategy Pattern in Pi

## Discovery Summary

Discovery agents scanned the Pi extension system (`docs/extensions.md`, `docs/sdk.md`, `docs/custom-provider.md`, `docs/models.md`), the built-in subagent example (`examples/extensions/subagent/`), the handoff extension (`examples/extensions/handoff.ts`), the preset extension (`examples/extensions/preset.ts`), and the tool registration pipeline (`pi-coding-agent/dist/core/`). Key files were read for depth: `subagent/index.ts` (the full subprocess-based subagent tool), `subagent/agents.ts` (agent discovery with model frontmatter), `handoff.ts` (in-process `complete()` call pattern), `preset.ts` (model switching via `pi.setModel()`), and the type definitions for `complete()`/`completeSimple()`, `ModelRegistry`, `AgentToolResult`, `Context`, and `AssistantMessage`. No existing docs or decisions about the advisor strategy pattern were found in the codebase — the closest concepts are per-agent `model:` frontmatter in subagent definitions and the `model?: string` parameter on the `Agent` tool.

## Questions

1. **Trace the in-process LLM call pattern for an advisor tool** — The `handoff.ts` example at `examples/extensions/handoff.ts` demonstrates making a direct LLM call from within an extension by importing `complete` from `@mariozechner/pi-ai`, resolving auth via `ctx.modelRegistry.getApiKeyAndHeaders(ctx.model!)`, and passing `{ apiKey, headers, signal }` to the completion options. Trace how this same pattern could be used to build a custom `advisor` tool that: receives a question from the executor (e.g., Sonnet), resolves a different model (e.g., Opus) via `ctx.modelRegistry.find("anthropic", "claude-opus-4-6")`, calls `complete()` with a curated advisor system prompt, and returns the advisor's guidance as the tool result `content`. Show exactly how to construct the `Context` object (systemPrompt + messages) for the advisor model and how the response text flows back to the executor. This matters because it's the core mechanism for implementing the advisor strategy without subprocess spawning.

2. **Trace the `complete()` / `completeSimple()` function signatures and return types** — From the type definitions in `@mariozechner/pi-ai/dist/stream.d.ts`, trace how `complete<TApi>(model, context, options?)` and `completeSimple<TApi>(model, context, options?)` differ: which one accepts `SimpleStreamOptions` with `reasoning`/`thinkingBudgets`, which accepts `ProviderStreamOptions` with `onPayload`. Map the `Context` type (`{ systemPrompt?, messages, tools? }`), the `AssistantMessage` return type (with `content: (TextContent | ThinkingContent | ToolCall)[]`, `usage`, `stopReason`), and how to extract the text response from `content.filter(c => c.type === "text").map(c => c.text).join("\n")`. This matters because the advisor tool must correctly call the API and parse the response.

3. **Trace the model resolution and auth pipeline for calling a different model** — From `ctx.modelRegistry.find("anthropic", "claude-opus-4-6")` which returns a `Model<Api> | undefined`, through `ctx.modelRegistry.getApiKeyAndHeaders(advisorModel)` which returns `ResolvedRequestAuth = { ok: true, apiKey?, headers? } | { ok: false, error: string }`, to passing `auth.apiKey` and `auth.headers` as `SimpleStreamOptions` to `completeSimple()`. Show what happens when the executor is using one provider (e.g., OpenAI) but the advisor needs to use a different provider (e.g., Anthropic) — does `getApiKeyAndHeaders()` resolve keys from the correct provider? This matters because the advisor must authenticate independently from the executor.

4. **Trace how `before_provider_request` could inject the native Anthropic `advisor_20260301` tool** — From the `before_provider_request` event handler in `extensions/types.d.ts` which receives `{ type: "before_provider_request", payload: unknown }` and can return a mutated payload, through the `onPayload` callback in `sdk.js` which chains handler results, to the actual HTTP body sent to the provider. Show whether an extension could detect Anthropic payloads and inject `{ type: "advisor_20260301", name: "advisor", model: "claude-opus-4-6", max_uses: 3 }` into the `tools` array of the raw JSON body. This matters because it would let Pi use Anthropic's server-side advisor tool natively, where the model handoff happens inside a single `/v1/messages` request with no extra round-trips.

5. **Trace the subagent subprocess pattern and compare with the in-process `complete()` pattern** — From the `subagent/index.ts` extension which spawns a separate `pi` process via `spawn()` with `--mode json -p --no-session`, parses JSONL events from stdout, and collects `AssistantMessage` results — versus the `handoff.ts` extension which calls `complete()` directly in-process. Compare: context isolation (subprocess gets fresh context vs. in-process shares parent context), cost (subprocess has its own session overhead), latency (subprocess startup time vs. direct API call), and tool access (subprocess has full Pi tool suite vs. in-process has no tools). This matters because the advisor pattern specifically requires the advisor to NOT call tools — so the simpler in-process `complete()` approach is a better fit.

6. **Trace the tool registration and prompt serialization pipeline for a custom advisor tool** — From `pi.registerTool({ name: "advisor", ... })` in an extension, through `loader.js` which stores it in `extension.tools.set()`, through `agent-session.js:1774 _refreshToolRegistry()` which merges extension tools with built-in and SDK tools, through `system-prompt.js:7 buildSystemPrompt()` which includes `promptSnippet` in the "Available tools" section and `promptGuidelines` in the "Guidelines" section. Show how `promptSnippet` and `promptGuidelines` on the advisor tool definition would instruct the executor model (e.g., Sonnet) about when and how to invoke the advisor. This matters because the LLM must know to call the advisor tool when it encounters a difficult decision.

7. **Trace how the `@tintinweb/pi-subagents` library's `model` frontmatter field and the built-in subagent example's per-agent `model:` frontmatter work** — From agent `.md` files with `model: claude-haiku-4-5` in YAML frontmatter, through `agents.ts discoverAgents()` which parses frontmatter into `AgentConfig.model`, through `subagent/index.ts` which passes `--model agent.model` to the spawned `pi` process. Show how this per-agent model selection could be adapted for the advisor pattern — where the advisor agent definition would specify `model: claude-opus-4-6` while the executor runs on a cheaper model. This matters because it's an existing, proven pattern for multi-model dispatch that could be leveraged.

8. **Trace the full data flow for an advisor tool execution with context curation** — From the executor (Sonnet) calling the `advisor` tool with `{ question: "...", context_summary: "..." }`, through the tool's `execute()` method which: (a) gets the current conversation branch via `ctx.sessionManager.getBranch()`, (b) converts messages to LLM format via `convertToLlm()` from `@mariozechner/pi-coding-agent`, (c) constructs a `Context` with an advisor system prompt + curated messages, (d) resolves the advisor model's API key via `ctx.modelRegistry.getApiKeyAndHeaders()`, (e) calls `complete()` with the advisor model, and (f) returns the advisor's response as `AgentToolResult`. Show how `convertToLlm()` transforms session entries into the `Message[]` format expected by `complete()`. This matters because context curation (what to send to the advisor) is critical for both cost and quality.
