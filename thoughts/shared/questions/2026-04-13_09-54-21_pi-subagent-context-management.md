# Research Questions: pi-subagent context management for weak models (GLM-4.7)

## Discovery Summary

rpiv-pi's subagent runtime lives entirely in the peer dependency `@tintinweb/pi-subagents@0.5.2` (installed at `~/.nvm/versions/node/v25.1.0/lib/node_modules/@tintinweb/pi-subagents/`); rpiv-pi itself only ships 9 bundled agent `.md` profiles under `agents/`, auto-copied at session start by `extensions/rpiv-core/agents.ts:copyBundledAgents`. The dispatch path is `Agent`-tool `execute()` → `AgentManager.spawn()` → `runAgent()` (agent-runner.ts) → `createAgentSession({ sessionManager: SessionManager.inMemory(cwd) })` per call — every subagent is a fresh, independent session with its own full-window context; no cross-session context summarization or output truncation exists. Eleven files were read for depth: `pi-subagents/src/{default-agents.ts, prompts.ts, agent-runner.ts, agent-manager.ts, context.ts, invocation-config.ts, index.ts}`, `rpiv-pi/extensions/rpiv-core/{index.ts, agents.ts, package-checks.ts, templates/pi-permissions.jsonc}`, `rpiv-pi/agents/{codebase-analyzer.md, codebase-locator.md}`, and `rpiv-pi/skills/research/SKILL.md`, plus the pi-subagents CHANGELOG (v0.4.0 removed the 50-turn default cap — unlimited is now the default). The overall shape: subagent context bloat with a weak model (GLM-4.7 configured in `~/.pi/agent/settings.json`) stems from the multiplicative product of (a) unlimited `max_turns`, (b) `append`-mode parent-system-prompt inheritance, (c) re-injected guidance files inside subagent sessions, (d) full extension-tool schemas shipped every turn, and (e) verbose agent/skill prompt instructions that push models to "trace exhaustively."

## Questions

1. Trace the default turn budget — why unlimited is the load-bearing default. Follow the turn-counting path from `pi-subagents/src/agent-runner.ts:312` (`normalizeMaxTurns(options.maxTurns ?? agentConfig?.maxTurns ?? defaultMaxTurns)`) through the soft-limit steer in the `turn_end` subscriber at `agent-runner.ts:317-329` (emits *"You have reached your turn limit — wrap up immediately…"* then allows `graceTurns=5` more before `session.abort()`), the module-level `defaultMaxTurns` initialized to `undefined` at `agent-runner.ts:28` (explicitly commented "unlimited = no turn limit"), and confirmed by CHANGELOG v0.4.0 ("subagents no longer have a 50-turn default cap"). Show how `resolveAgentInvocationConfig()` in `invocation-config.ts:30` lets frontmatter `maxTurns` or Agent-tool `max_turns` param override it, and identify where a rpiv-pi-wide default could be set (e.g. `setDefaultMaxTurns()` invoked from `extensions/rpiv-core/index.ts` at init, or `/agents` → Settings). This matters because with GLM-4.7 and no cap, a single `codebase-analyzer` run can loop dozens of turns, each re-feeding the full growing conversation back to the model — the single biggest driver of the 200k-token blow-up.

2. Trace the system-prompt inheritance chain for `general-purpose` and every rpiv-pi agent dispatch. Starting at `default-agents.ts:11-28` where `general-purpose` is defined with `promptMode: "append"` and an empty `systemPrompt`, follow `runAgent()` calling `ctx.getSystemPrompt()` at `agent-runner.ts:169` to pull the parent's full prompt, then `buildAgentPrompt()` in `prompts.ts:49-70` which wraps it in `<inherited_system_prompt>…</inherited_system_prompt>` plus a `<sub_agent_context>` bridge and any `<agent_instructions>` body. In rpiv-pi specifically, the inherited parent prompt already carries CLAUDE.md content, injected guidance sections, tool registration blurbs for `ask_user_question`/`todo`/`advisor`, and the Pi framework preamble — all duplicated verbatim for every subagent call, every turn. Explain how switching a bundled rpiv-pi agent's frontmatter to `prompt_mode: replace` (the mode used by `Explore`/`Plan` in `default-agents.ts:67, 123`) drops this entire inheritance and saves kilobytes of fixed prompt per subagent turn. This matters because weak models are both thrown off by large system prompts AND pay the inherited-prompt token cost on every turn.

3. Trace the guidance-injection re-entry into subagent sessions. Walk from `extensions/rpiv-core/index.ts:135-137` (`pi.on("tool_call", handleToolCallGuidance)`) into `guidance.ts` — per the root CLAUDE.md it "resolves per-depth at most one of `AGENTS.md > CLAUDE.md > .rpiv/guidance/<sub>/architecture.md`" and injects each new file via `pi.sendMessage({ display: false })` with an in-process `Set` deduping across the session and cleared on `session_start`/`session_compact`/`session_shutdown`. Then show that `agent-runner.ts:299` calls `session.bindExtensions()` on the fresh subagent session (`SessionManager.inMemory(effectiveCwd)` at `agent-runner.ts:256`), which — per pi-subagents CHANGELOG 0.5.2 — re-fires the rpiv-core extension's `session_start` and re-wires the `tool_call` listener. Because the dedup `Set` is session-scoped, every subagent re-injects every guidance file its tools touch. Quantify the effect: a locator that greps across 4 subfolders pulls in 4 architecture.md files, each re-sent on every subsequent turn's context. This matters because guidance doubles as a hidden context amplifier inside subagents — the fix likely requires either a cross-session dedup registry keyed on `(cwd+depth)` or skipping guidance injection for tool calls whose `depth > 0` originated inside a subagent session.

4. Explain how tool inheritance works and what `isolated` / `disallowed_tools` / `tools` prune. Trace `getToolsForType()` at `agent-runner.ts:186` and the active-tool filter at `agent-runner.ts:277-293`: `builtinToolNames` from the `tools:` frontmatter gates built-ins, `extensions: true` (default, not set by any rpiv-pi agent — see `agents/CLAUDE.md`) admits ALL extension/MCP tools, and `EXCLUDED_TOOL_NAMES = ["Agent", "get_subagent_result", "steer_subagent"]` at `agent-runner.ts:25` is the only automatic removal. Show that every rpiv-pi locator therefore inherits `ask_user_question`, `todo`, `advisor`, `web_search`, `web_fetch` plus any MCP tools, bloating the JSON tool schemas sent every turn. Explain how `isolated: true` (consumed in `invocation-config.ts:33` and branched in `agent-runner.ts:174-176` to force `extensions=false`/`skills=false`) collapses the set to built-ins only, and how `disallowed_tools:` (`agent-runner.ts:271-293`, parsed in `custom-agents.ts`) trims specific names. This matters because tool-schema JSON is resent every turn — weak models both waste input tokens AND get distracted into spurious tool calls when the tool menu is too large.

5. Trace how the rpiv-pi bundled agents dispatch today vs the available knobs they don't set. Read `rpiv-pi/agents/codebase-analyzer.md`, `codebase-locator.md`, and the full frontmatter schema documented at `@tintinweb/pi-subagents/README.md:149-174`. Show that none of the 9 bundled rpiv-pi agents specify `model`, `thinking`, `max_turns`, `prompt_mode`, `isolated`, `disallowed_tools`, `inherit_context`, or `extensions: false` — so every dispatch from `skills/research/SKILL.md:50-88` and peers inherits parent model (GLM-4.7), parent thinking level (`"high"` per `~/.pi/agent/settings.json`), unlimited turns, `append`-mode parent-twin prompt, and all extension tools. Map each frontmatter field to the code path that consumes it (`custom-agents.ts` parsers → `AgentConfig` shape in `types.ts` → `resolveAgentInvocationConfig` in `invocation-config.ts:13-36` → `runAgent` in `agent-runner.ts:154-368`). This matters because the immediate, high-leverage fix may be a 5–10-line frontmatter edit across the 9 bundled agents in `rpiv-pi/agents/*.md`, not a code change — and the auto-copy flow at `extensions/rpiv-core/agents.ts:36-62` will propagate those edits to `.pi/agents/` on every session start.

6. Explain the skill-level prompt pressure that drives long agent outputs. Read `skills/research/SKILL.md` Step 2 (agent prompt template at lines 56-82) and `skills/research-questions/SKILL.md` locator prompts — both instruct agents to "Focus on DEPTH — trace the actual code" and pass dense multi-paragraph questions verbatim as prompts. Cross-reference with the `## Output Format` blocks in `agents/codebase-analyzer.md:49-101` and `codebase-locator.md:56-86`, both of which ship long, verbose realistic-example output blocks that weak models pattern-match and expand on. Confirm there is no agent-prompt chunking or length cap at the skill layer. This matters because output length is bounded only by what the model self-terminates on — GLM-4.7 with a "be exhaustive, trace everything" directive plus a ~80-line example output has no off-switch, so part of the fix is rewriting skill prompt templates and agent Output-Format blocks to cap verbosity (e.g. "Return ≤X bullets, no prose, no code blocks > 3 lines, one section per area").

7. Trace how concurrency, background join, and foreground dispatch interact with per-agent context size. Walk the `AgentManager.spawn()` path at `agent-manager.ts:85-115` and `DEFAULT_MAX_CONCURRENT=4` (`agent-manager.ts:20`), `startAgent()` kicking off `runAgent()` at `agent-manager.ts:140-166`, and the `run_in_background` routing in `pi-subagents/src/index.ts:553+` Agent-tool `execute()`. Show that each concurrent agent is a fully independent `AgentSession` with its own full-window context — 4 parallel `codebase-analyzer` calls at 50k each ≈ 200k peak memory PLUS each returns its full result string into the parent via `get_subagent_result` or the foreground return path (`index.ts:131-134` only truncates the notification *preview* at `resultMaxLen=500/300`; `record.result` itself is uncapped). This matters because rpiv-pi's `skills/research` "spawn 3–6 parallel analyzers" pattern multiplicatively inflates both per-agent AND parent-side context on join — weak models can't digest six 10k-token analyst reports glued into one turn, so the fix must cap per-agent output AND consider sequential-with-summarization dispatch for weak-model mode.

8. Trace `inherit_context` and `buildParentContext` — the optional megabomb path. Read `context.ts:20-57` and follow the wire-up at `agent-runner.ts:350-356` where `options.inheritContext` prepends the entire serialized parent conversation (user + assistant text, skipping toolResults and including compaction summaries) as a prefix to the agent's prompt. Show that `inheritContext` defaults to `false` at `invocation-config.ts:31` (safe), but any rpiv-pi skill or custom agent setting it true causes every subagent turn to re-send the parent's history, on top of the already inherited system prompt. Audit the 9 bundled agents and all `skills/*/SKILL.md` files to confirm none set it, and propose a repo-level guardrail (a frontmatter lint or a rpiv-core warning) to keep it off. This matters because any accidental `inherit_context: true` addition will instantly multiply subagent context by the parent's running size — it must stay off AND be documented as a no-go for weak-model setups.

9. Identify all levers to cap subagent response/result size on the return path. Starting from `agent-runner.ts:119-130` `collectResponseText()` and `getLastAssistantText()` (both return full last-message text, no truncation), through `index.ts:131-134` `formatTaskNotification` (only the *preview* is truncated at `resultMaxLen=500/300` — full result still lives on `record.result`), and `get_subagent_result` which returns `record.result` unbounded (`index.ts` near the `get_subagent_result` tool registration). Show there is no `maxOutputTokens` plumbed through `createAgentSession`, no response-size cap in the Agent tool registration at `index.ts:553-629`, and no in-band truncation in the per-turn loop in `agent-runner.ts:317-344`. The only backpressure mechanisms are `max_turns` + `graceTurns` (budget, not size) and the agent's own self-limiting output instructions. This matters because once a turn emits, the full delta is in the subagent's own context AND will be copied into the parent on completion — the cap has to live either at `max_turns` (turn budget) or in the agent's `## Output Format` spec (self-limiting instructions via prompt engineering), since no upstream knob exists.

10. Map the concrete rpiv-pi-side improvement surface end-to-end. Identify every edit site that could be changed without modifying `@tintinweb/pi-subagents`: (a) add `max_turns`, `thinking: low`, `model: <cheap-locator-model>`, `isolated: true`, `disallowed_tools`, `prompt_mode: replace` frontmatter to each `rpiv-pi/agents/*.md` — auto-copied via `extensions/rpiv-core/agents.ts:36-62`; (b) call `setDefaultMaxTurns(N)` and `setGraceTurns(M)` from `extensions/rpiv-core/index.ts` at init — requires importing from `@tintinweb/pi-subagents` (which is a peer dep), so evaluate the tradeoff vs a `/rpiv-configure-agents` command that mutates pi-subagents' `/agents → Settings`; (c) tighten `guidance.ts` so the dedup `Set` keys on `(cwd+depth)` globally or skips injection for subagent-origin tool calls, avoiding re-injection of files the parent already delivered; (d) tighten bundled-agent `## Output Format` blocks in `agents/*.md` to explicitly cap bullet counts and forbid verbose expansion (also documented in `agents/CLAUDE.md` `<important if="adding a new agent">` block); (e) rewrite skills' agent-prompt templates (`skills/research/SKILL.md:60-82`, `skills/research-questions/SKILL.md` locator prompts, plus design/write-plan/code-review prompts that spawn agents) to include `"Return ≤N bullets, no prose, no code blocks > 3 lines"`. For each site, identify which rpiv-pi file changes, who consumes it, and whether the change is backward-compatible. This matters because it is the deliverable: a prioritized punch-list of surgical rpiv-pi edits that reduce subagent context for weak models like GLM-4.7 without forking the upstream package.
