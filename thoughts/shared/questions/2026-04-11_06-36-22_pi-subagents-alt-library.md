# Research Questions: nicobailon/pi-subagents vs @tintinweb/pi-subagents

## Discovery Summary

rpiv-pi's subagent layer is intentionally minimal: **no runtime dispatcher code, no imports from the subagent package anywhere in TypeScript**. It consists of (1) 62 lines of TS in `extensions/rpiv-core/agents.ts` that `copyFileSync` 9 bundled agent `.md` files into `<cwd>/.pi/agents/` on `session_start`, (2) 9 markdown prompt files in `agents/` with a three-key frontmatter schema (`name`, `description`, `tools` as a lowercase CSV), (3) a permissions template at `extensions/rpiv-core/templates/pi-permissions.jsonc:27-29` that hard-codes `Agent`, `get_subagent_result`, `steer_subagent` as allowed tool names, (4) a `before_agent_start` hook at `extensions/rpiv-core/index.ts:108-125` that injects git context into every subagent via a `customType: "rpiv-git-context"` message, and (5) 16 SKILL.md consumers under `skills/` (~131 literal references) that dispatch via LLM prose like `subagent_type: codebase-locator`. The runtime engine is `@tintinweb/pi-subagents@^0.5.2`, declared at `package.json:18`, which registers the `Agent` tool, scans `<cwd>/.pi/agents/*.md`, and runs subagents in-process via `createAgentSession`.

`nicobailon/pi-subagents` (npm: `pi-subagents`, 702⭐, 116 commits, 16 open issues) is a **superset** library from a per-README scan: it registers a differently-named tool (`subagent`, parameters `{agent, task}` not `{subagent_type, prompt}`), scans the same `.pi/agents/*.md` path plus `.chain.md`, accepts the same three frontmatter keys *plus* `extensions`, `model`, `thinking`, `skill`, `output`, `defaultReads`, `defaultProgress`, `interactive`, and `maxSubagentDepth`, spawns agents as **separate OS processes** via `pi-spawn` (vs. `@tintinweb`'s in-process execution), ships 7 builtin agents (scout, planner, worker, reviewer, context-builder, researcher, delegate), and adds features like sequential `.chain.md` files, async background execution tracked via `subagent_status`, parallel `{tasks: [...]}` with `concurrency`/`failFast`, `worktree: true` per-agent git isolation, and XML-wrapped SKILL.md injection into subagent system prompts.

Key files read for depth: `extensions/rpiv-core/agents.ts`, `extensions/rpiv-core/index.ts`, `extensions/rpiv-core/package-checks.ts`, `extensions/rpiv-core/permissions.ts`, `extensions/rpiv-core/templates/pi-permissions.jsonc`, `package.json`, `agents/codebase-locator.md`, `agents/web-search-researcher.md`. Zero prior thoughts/ docs evaluate `nicobailon/pi-subagents` — all historical agent-library discussion in `thoughts/shared/research/2026-04-10_13-45-00_complete-pi-migration.md` and `thoughts/shared/designs/2026-04-10_11-18-29_complete-pi-migration.md` is about `@tintinweb/pi-subagents` and the rejected in-tree `@mariozechner/pi-coding-agent` example.

## Questions

1. **Tool-name and parameter-name rewrite surface.** Trace every place in rpiv-pi that hardcodes the literal string `"Agent"` (tool name) and `subagent_type` (parameter name) — from `extensions/rpiv-core/templates/pi-permissions.jsonc:27-29` (allow-list entries `Agent`, `get_subagent_result`, `steer_subagent`), through the 16 SKILL.md files under `skills/*/SKILL.md` that invoke subagents via prose (e.g., `skills/annotate-guidance/SKILL.md:28,32,83,87`, `skills/annotate-inline/SKILL.md:28,32,81,85`, and `skills/write-test-cases/SKILL.md:80,87,97,101`), to the `/rpiv-setup` command handler at `extensions/rpiv-core/index.ts:154-158` whose reason string names those three tool names. `nicobailon/pi-subagents` registers a tool called `subagent` with parameters `{agent, task}` instead. Determine the exact rewrite cost: how many literal occurrences of `Agent`, `subagent_type`, `get_subagent_result`, `steer_subagent` exist across rpiv-pi, which are load-bearing (actually executed by the LLM as tool calls or regex-matched at runtime) vs. cosmetic (docs/warnings/reason strings), and whether a mechanical search-and-replace is sufficient or the skills must be restructured because the payload shapes differ.

2. **Agent-discovery path compatibility and name collisions with builtin agents.** Both libraries scan `<cwd>/.pi/agents/*.md`, and `copyBundledAgents(cwd, overwrite)` at `extensions/rpiv-core/agents.ts:36-62` writes exactly there (line 46: `const targetDir = join(cwd, ".pi", "agents")`, line 49: filter `.md`), so the copy function should work unchanged. Verify this claim against `nicobailon/pi-subagents`'s declared discovery scopes (project `.pi/agents/{name}.md`, user `~/.pi/agent/agents/{name}.md`, builtin `~/.pi/agent/extensions/subagent/agents/`) and confirm that the 9 bundled names (`codebase-locator`, `codebase-analyzer`, `codebase-pattern-finder`, `integration-scanner`, `precedent-locator`, `test-case-locator`, `thoughts-locator`, `thoughts-analyzer`, `web-search-researcher`) will not collide with or be shadowed by the library's 7 builtin agents (scout, planner, worker, reviewer, context-builder, researcher, delegate). Also verify the `agentScope` default resolution order — project agents must override user/builtin, otherwise `copyBundledAgents` becomes a no-op.

3. **Frontmatter schema compatibility and implicit capability changes.** The 9 bundled agents declare exactly three keys — `name`, `description`, and `tools` (lowercase CSV like `grep, find, ls` in `agents/codebase-locator.md:1-5` or `web_search, web_fetch, read, grep, find, ls` in `agents/web-search-researcher.md:1-5`). Trace how `nicobailon/pi-subagents`'s frontmatter parser treats omitted fields: per its README, `extensions:` absent means "all extensions load" (permissive) while empty means "none" (restrictive); `thinking:`, `model:`, and `maxSubagentDepth:` defaults are unknown. Determine whether omitting these fields preserves today's behavior under `@tintinweb/pi-subagents`, or whether defaults silently grant or restrict capabilities — e.g., will `thoughts-analyzer` still be read-only, will `precedent-locator` still get `bash` access (the only bundled agent that lists `bash` at `agents/precedent-locator.md:4`), and will `web-search-researcher` still reach the extension-registered `web_search`/`web_fetch` tools from `extensions/web-tools/index.ts:170,288`?

4. **`before_agent_start` lifecycle hook under subprocess execution.** rpiv-core's git-context injection lives at `extensions/rpiv-core/index.ts:108-125` and depends entirely on the Pi lifecycle event `before_agent_start` firing inside every subagent spawn, returning a `customType: "rpiv-git-context"` message with `display: false`. `@tintinweb/pi-subagents` runs subagents in-process so the event propagates through the shared extension registry. `nicobailon/pi-subagents` spawns subagents as **separate OS processes** via `pi-spawn` with optional `context: "fork"` vs `"fresh"` modes. Investigate whether a spawned child process can receive `before_agent_start` events, whether rpiv-core's extension is even loaded inside the child, and what happens to the `pi.appendEntry("active_agent", { name: "general-purpose" })` workaround at `index.ts:45` that exists solely to unblock `pi-permission-system@0.4.1`'s input handler — if the child process doesn't inherit session entries, both the git-context injection and the permission-system shim break silently.

5. **Custom rpiv-pi tool inheritance across the subprocess boundary.** `extensions/rpiv-core/ask-user-question.ts` registers `ask_user_question` and `extensions/rpiv-core/todo.ts` registers `todo` via `pi.registerTool(...)`. Under the current library, subagents inherit these because execution is in-process. The permissions template allows them at `pi-permissions.jsonc:25-26`, but the 9 agent frontmatters do **not** list them in their `tools:` CSV (they list only builtins: `read, grep, find, ls, bash, web_search, web_fetch`). Under `nicobailon/pi-subagents`, the `extensions:` frontmatter field controls which extension `.ts` files a spawned child loads — absent = all, empty = none, CSV = allowlist. Determine (a) whether spawned children automatically load `extensions/rpiv-core/index.ts` and re-register `ask_user_question`/`todo`, (b) whether the agents' `tools:` CSV must explicitly list these custom tool names to grant access, and (c) whether `todo` state reconstruction via `reconstructTodoState(ctx)` at `index.ts:35,99` still works if each subagent has its own process-local state.

6. **Silent `general-purpose` fallback and agent-name resolution.** The current library has a documented silent fallback at `@tintinweb/pi-subagents/src/index.ts:730-732` (referenced in `thoughts/shared/plans/2026-04-10_12-46-17_complete-pi-migration.md:1388`) where an unknown `subagent_type` resolves to a builtin `general-purpose` agent. rpiv-core relies on this in two places: `extensions/rpiv-core/index.ts:45` seeds `pi.appendEntry("active_agent", { name: "general-purpose" })` as a `pi-permission-system@0.4.1` workaround, and `skills/validate-plan/SKILL.md:52-54` plus `skills/resume-handoff/SKILL.md:48` literally say "spawn parallel general-purpose agents via the Agent tool". Investigate how `nicobailon/pi-subagents` handles an unknown `agent` parameter — does it error, fall back to a builtin like `delegate` or `worker`, or require a user-authored `general-purpose.md`? The answer determines whether `validate-plan` and `resume-handoff` break on day one, whether the permission-system shim at `index.ts:45` still works, and whether an rpiv-pi-bundled `general-purpose.md` stub must be added to `agents/`.

7. **Skill-consumer invocation conventions and rewrite scope across 16 files.** The 16 skills that invoke agents (full list: `annotate-guidance`, `annotate-inline`, `code-review`, `create-plan`, `design-feature`, `design-feature-iterative`, `evaluate-research`, `iterate-plan`, `outline-test-cases`, `research`, `research-codebase`, `research-questions`, `research-solutions`, `resume-handoff`, `validate-plan`, `write-test-cases`) use two prose patterns: inline YAML `- subagent_type: \`codebase-locator\`` (in the three annotate/write-test-cases skills) and free-prose "Use the Agent tool with codebase-locator" (everywhere else). Count total literal references (prior discovery reports ~131 across 16 files), then trace how `nicobailon/pi-subagents`'s `subagent` tool schema expects payloads — single `{agent, task}`, sequential `{chain: [...]}`, parallel `{tasks: [...]}` — and judge whether each skill's parallel fan-out pattern (e.g., `skills/research-codebase/SKILL.md:41-54` spawns 8 agents; `skills/evaluate-research/SKILL.md:151-191` spawns 7 with explicit prompt templates) translates 1-to-1 to `tasks: [...]` or requires rewriting the skill's step structure to align with the new tool's concurrency model.

8. **Package-detection, install command, and permission-allow-list rewrites.** rpiv-pi hard-codes the current library's npm name in four load-bearing places: `package.json:18` (`@tintinweb/pi-subagents: ^0.5.2` in `dependencies`), `extensions/rpiv-core/package-checks.ts:34` (regex `/@tintinweb\/pi-subagents/i` against `~/.pi/agent/settings.json` → `packages[]`), `extensions/rpiv-core/index.ts:81` (the `session_start` warning string shown via `ctx.ui.notify`), and `extensions/rpiv-core/index.ts:155-158` (the `/rpiv-setup` command that runs `pi install npm:@tintinweb/pi-subagents`). `nicobailon/pi-subagents` ships as `pi install npm:pi-subagents`. Map each occurrence to its required rewrite, determine whether the permissions template at `pi-permissions.jsonc:27-29` must **remove** the old allow-list entries to avoid a latent permission leak into a different tool namespace, and verify what new allow-list keys are needed (`subagent`, `subagent_status`, any background-run tools, and the `/run`/`/chain`/`/parallel`/`/subagents-status`/`/agents` slash commands).

9. **New capabilities — chains, parallel, async, worktrees, skill injection — mapped to existing skill workflows.** `nicobailon/pi-subagents` exposes features absent from the current library: sequential `.chain.md` files with per-step `output`/`reads`/`model`/`skills`/`progress` config, `{tasks: [...]}` parallel arrays with configurable `concurrency` and `failFast`, async background execution via `async: true` tracked by `subagent_status`, `worktree: true` per-agent git isolation, and XML-wrapped SKILL.md injection (`<skill name="...">...</skill>`) into subagent system prompts. Evaluate which existing rpiv-pi workflows would collapse or improve: could `skills/research-codebase/SKILL.md`'s 8-inline-Agent fan-out at lines 41-54 become a single `{tasks: [...]}` call with `concurrency: 8`, could `skills/research/SKILL.md`'s "always-spawn precedent-locator sidecar" pattern at line 85 become an `async: true` background job, could `skills/create-plan/SKILL.md` compose phases 1 and 2 as a `.chain.md` file, and would `worktree: true` let `skills/evaluate-research` run A-branch and B-branch verification against actual git worktrees? Identify which of the 16 skills gain material value vs. none.

10. **Pi-SDK version pinning, license, and maintenance-velocity compatibility.** The current `@tintinweb/pi-subagents` is declared as `^0.5.2` at `package.json:18` and is referenced in the migration design at `thoughts/shared/designs/2026-04-10_11-18-29_complete-pi-migration.md:45-50` with an MIT LICENSE note. For `nicobailon/pi-subagents`, determine: license file and terms, repository activity (per GitHub page: 702 stars, 91 forks, 116 commits, 16 open issues), Pi SDK pinning (what range of `@mariozechner/pi-coding-agent` does it peer-depend on, and does that conflict with rpiv-pi's own peer-deps at `package.json:11-16` which declare `@mariozechner/pi-ai: "*"`, `@mariozechner/pi-coding-agent: "*"`, `@mariozechner/pi-tui: "*"`, `@sinclair/typebox: "*"`?), the stability of its `subagent` tool schema between v0.x releases (breaking-change history), whether its release cadence is compatible with rpiv-pi's v0.2.0 pace, and any notable bug patterns from its 16 open issues that would affect rpiv-pi's usage (e.g., the MCP first-run caching caveat, interactive TUI gaps in v1, builtin-override semantics).
