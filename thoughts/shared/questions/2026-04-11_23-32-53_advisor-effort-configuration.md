# Research Questions: Advisor Effort Configuration

## Discovery Summary

Discovery agents mapped the full cross-provider handling of `SimpleStreamOptions.reasoning` across all 12 pi-ai providers (anthropic, openai-responses, openai-completions, openai-codex-responses, azure-openai-responses, google, google-vertex, google-gemini-cli, mistral, amazon-bedrock, faux, plus shared helpers in simple-options.js and models.js). A second agent traced `ThinkingSelectorComponent` in pi-coding-agent — a 51-line public export at `modes/interactive/components/thinking-selector.js` that takes `(currentLevel, availableLevels, onSelect, onCancel)` with all capability filtering done upstream by `agent-session.js:1161` `getAvailableThinkingLevels()`. A third agent catalogued 6 multi-step selector precedents across pi-subagents, rpiv-core, and pi-permission-system, establishing sequential `await ctx.ui.custom()` with `if (!choice) return;` as the idiomatic pattern. A fourth agent confirmed that the original advisor artifact chain (research, design, plan) chose `completeSimple()` specifically for its `reasoning?: ThinkingLevel` support but never passes a value — the gap was never raised as scope or explicitly deferred; it is a silently-unfilled slot.

## Questions

1. Trace the exact code path when the advisor tool in `extensions/rpiv-core/advisor.ts:148-152` calls `completeSimple(advisor, context, { apiKey, headers, signal })` with no `reasoning` field, through the pi-ai dispatch in `stream.js:19-21` (`streamSimple` → `provider.streamSimple`), into each provider's `streamSimple*` function. For Anthropic (`anthropic.js:381-389`, `streamSimpleAnthropic`), the `!options?.reasoning` guard at line 387 forwards `thinkingEnabled: false`, which `buildParams` at line 512 translates to `params.thinking = { type: "disabled" }` (gated on `model.reasoning`). For OpenAI Responses (`openai-responses.js:100`), `reasoningEffort` resolves to `undefined`, and `buildParams` at line 156 falls through to `reasoning: { effort: "none" }` on reasoning-capable models. For Google (`google.js:222-224`), it forwards `thinking: { enabled: false }`, dispatched by `getDisabledThinkingConfig` at line 315 to either `thinkingBudget: 0` (Gemini 2.x) or `thinkingLevel: "LOW"/"MINIMAL"` (Gemini 3 Pro/Flash — cannot truly disable). For Mistral (`mistral.js:66-69`), `promptMode` becomes `undefined`. Confirm whether this "thinking disabled" default is acceptable for an advisor whose purpose is stronger judgment, or whether it effectively negates the value of choosing a reasoning-capable advisor model like Opus 4.6.

2. The advisor module state in `advisor.ts:66` is a single `let selectedAdvisor: Model<Api> | undefined` with `getAdvisorModel()`/`setAdvisorModel()` accessors at lines 68-74. To add effort, this needs a companion `let selectedAdvisorEffort: ThinkingLevel | undefined` (where `ThinkingLevel` from `@mariozechner/pi-ai/dist/types.d.ts:7` is `"minimal" | "low" | "medium" | "high" | "xhigh"`) plus `getAdvisorEffort()`/`setAdvisorEffort()` accessors, and the `executeAdvisor` function at line 93 needs to read it and pass `reasoning: getAdvisorEffort()` in the `completeSimple()` options bag at line 151. Research: should the effort field also include `"off"` (matching `THINKING_LEVELS` in `agent-session.js:54` which includes `"off"`) to allow explicitly disabling thinking even on reasoning-capable advisors, or should `undefined` serve that role (matching the current behavior where omitting `reasoning` disables thinking)?

3. The `/advisor` command handler in `advisor.ts:284-400` currently opens a single `ctx.ui.custom<string | null>` panel at line 307 to pick a model. After the user picks a model, the handler should open a second panel for effort selection — following the sequential-await pattern established by pi-subagents' `showManualWizard` at `/usr/local/lib/node_modules/@tintinweb/pi-subagents/dist/index.js:1374` (6-step `ctx.ui.select` chain) and rpiv-core's own `ask-user-question.ts:60` (ctx.ui.custom → ctx.ui.input). Research: should the effort selector be a second `ctx.ui.custom` panel using an inline `SelectList` (matching the existing model-picker's style with `DynamicBorder`, prose header, themed `SelectList`), or should it reuse the exported `ThinkingSelectorComponent` from `@mariozechner/pi-coding-agent` (`dist/modes/interactive/components/thinking-selector.js:19`, constructor takes `(currentLevel, availableLevels, onSelect, onCancel)`) wrapped in a `ctx.ui.custom` call? Consider that `ThinkingSelectorComponent` uses `getSelectListTheme()` from pi-coding-agent's internal theme (not the `theme` parameter passed by `ctx.ui.custom`), which may produce visual inconsistency with the model-picker panel.

4. When the user picks an advisor model in the `/advisor` selector, the effort picker needs to offer only levels the picked model supports. The canonical capability check is `model.reasoning: boolean` (from `Model<Api>` in `@mariozechner/pi-ai/dist/types.d.ts:276-282`) gating whether thinking is supported at all, and `supportsXhigh(model)` (from `@mariozechner/pi-ai/dist/models.js:37-45`, checks for `gpt-5.2`/`gpt-5.3`/`gpt-5.4` or `opus-4-6`/`opus-4.6` in model.id) gating whether `"xhigh"` is available. The runtime `getAvailableThinkingLevels()` in `agent-session.js:1161-1164` returns `["off"]` for non-reasoning, `THINKING_LEVELS` (5 entries) for standard, or `THINKING_LEVELS_WITH_XHIGH` (6 entries) for xhigh-capable models — but this reads the *executor* model, not the advisor model. Research: should the advisor command import and call `supportsXhigh()` directly on the picked advisor model to build its own filtered level list, or is there a way to query `getAvailableThinkingLevels()` for an arbitrary model that is not the current session model?

5. When the user selects a new advisor model via the `/advisor` selector, what default effort should be applied? Options: (A) inherit the executor's current thinking level via `pi.getThinkingLevel()` (from `ExtensionAPI` at `types.d.ts:780-781`), so the advisor runs at the same effort as the main model; (B) hardcode `"high"` as the default for any reasoning-capable model, since the advisor's purpose is stronger judgment; (C) hardcode `"medium"` as a cost-balanced default; (D) `undefined` (no effort → thinking disabled), preserving backward compatibility. Additionally, when the user changes the advisor model (picks a different model without re-visiting effort), should the effort reset to the default or carry over from the prior model? The carry-over approach risks applying `"xhigh"` from an Opus advisor to a newly-picked Sonnet advisor where xhigh is not supported (would be clamped to `"high"` by `clampReasoning` in `simple-options.js:15-16`).

6. Trace the cost implications of setting `reasoning: "high"` on the advisor's `completeSimple()` call. For Anthropic Opus 4.6, `streamSimpleAnthropic` at `anthropic.js:392-398` maps `"high"` via `mapThinkingLevelToEffort` at lines 365-380 to effort `"high"`, which becomes `thinking: { type: "adaptive" }` at line 516 of `buildParams`. This triggers extended thinking tokens that are billed at output-token rates. Since the advisor receives the full serialized branch (the `conversationText` built at `advisor.ts:125-129` via `serializeConversation(convertToLlm(agentMessages))`) as a single user message, thinking cost scales with context size. For Google Gemini 2.5 Pro, `getGoogleBudget` at `google.js:350-373` maps `"high"` to a 32768-token thinking budget (`google.js:366`). For OpenAI, `streamSimpleOpenAIResponses` at `openai-responses.js:100-104` passes `reasoning: { effort: "high" }` directly. Research: given that the advisor always receives the full branch and each call is "deliberate rather than chatty" (per `ADVISOR_PROMPT_GUIDELINES` at `advisor.ts:238-245`), is `"high"` a reasonable default effort, or does the full-branch context already provide enough signal that `"medium"` would suffice — trading some reasoning depth for 2-4x lower thinking-token cost?

7. The `onUpdate` callback at `advisor.ts:142-145` currently emits `Consulting advisor (${advisorLabel})…` showing only the model provider:id. The `AdvisorDetails` interface at `advisor.ts:55-60` has fields for `advisorModel`, `usage`, `stopReason`, and `errorMessage` — no effort field. Research: should the effort level be surfaced in (a) the `onUpdate` status text (e.g. `"Consulting advisor (anthropic:claude-opus-4-6, high)…"`), (b) a new `effort?: ThinkingLevel` field on `AdvisorDetails` returned in every result (paralleling how `usage` and `stopReason` are always returned), and (c) the `ADVISOR_DESCRIPTION` at `advisor.ts:227-233` or `ADVISOR_PROMPT_SNIPPET` at line 235 so the executor model is aware of the advisor's configured effort level?
