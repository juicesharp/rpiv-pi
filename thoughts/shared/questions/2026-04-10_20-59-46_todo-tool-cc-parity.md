## Questions

1. **Data model & schema shape — how does the current `Todo { id: number; text: string; done: boolean }` interface at `/Users/sguslystyi/rpiv-pi/extensions/rpiv-core/index.ts:416-420` need to grow into a full `Task` record matching Claude Code's TaskCreate/TaskUpdate/TaskGet/TaskList schemas?** Trace the current TypeBox parameter schema at `rpiv-core/index.ts:455-459` (`action`, `text?`, `id?` via `StringEnum`/`Type.Object`/`Type.Optional`) against the Pi example's `TodoParams` at `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/examples/extensions/todo.ts:31-35`, the `TodoDetails` shape at `todo.ts:24-29`, and the already-drafted Option 2 TypeBox sketch at `/Users/sguslystyi/rpiv-skillbased/thoughts/shared/research/2026-04-10_gap-analysis-porting-to-pi.md:642-658`. Each added field (`subject`, `description`, `activeForm`, `status`, `blockedBy`, `blocks`, `owner`, `metadata`) must be serialised through the `AgentToolResult.details` payload defined at `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/node_modules/@mariozechner/pi-agent-core/dist/types.d.ts:248` because the reconstruction loop at `rpiv-core/index.ts:426-439` is the only persistence path — decide which fields are first-class vs stashed in `metadata`, and whether `subject` replaces or supplements the existing `text` field when old session histories are replayed.

2. **API shape — one action-discriminated tool vs four tools sharing closure state: should the upgrade keep rpiv-core's single-tool + `action` enum pattern at `/Users/sguslystyi/rpiv-pi/extensions/rpiv-core/index.ts:444-523` (mirrored in Pi's example `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/examples/extensions/todo.ts:136-281`), or register four tools named exactly `TaskCreate`/`TaskUpdate`/`TaskList`/`TaskGet` via four consecutive `pi.registerTool` calls that close over the same module-scope `tasks`/`nextId` variables the way `rpiv-core/index.ts:422-423` does today?** Both shapes are permitted by the `ToolDefinition<TParams, TDetails, TState>` interface at `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/dist/core/extensions/types.d.ts:281-302`; the four-tool path gives CC-identical tool names (letting the 12 rpiv-skillbased skills that still reference `TaskCreate`/`TaskUpdate` by name reuse their prose verbatim after porting), while the single-tool path minimises the TypeBox surface and matches the Pi idiom already proven in the repo. Neither approach is demonstrated by any other extension in `/Users/sguslystyi/rpiv-pi/extensions/` or in `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/examples/extensions/`, so this is the first decision that determines every downstream file and every skill-prose edit. Also weigh a hybrid: four thin delegating tools that share a single `execute` via a closure, which lets the `promptSnippet`/`promptGuidelines` at `types.d.ts:289-291` be tailored per-verb.

3. **Status transition semantics — how should the current boolean `done` toggle at `/Users/sguslystyi/rpiv-pi/extensions/rpiv-core/index.ts:495-508` become CC's `pending → in_progress → completed / deleted` state machine (the `TaskUpdate` schema's status enum: "Status progresses: pending → in_progress → completed; Use deleted to permanently remove a task")?** Trace the current promptGuidelines at `rpiv-core/index.ts:450-454` ("Use the todo tool (toggle action) to mark tasks as completed as you finish each step. This replaces TaskCreate/TaskUpdate from other systems.") against the only live skill that currently assumes status vocabulary: `/Users/sguslystyi/rpiv-pi/skills/resume-handoff/SKILL.md:177` ("Tasks marked as 'in_progress' in handoff"). Decide (a) whether `activeForm` (the present-continuous spinner string CC shows while a task is in_progress, e.g. "Running tests") becomes a persisted field or a transient render hint, (b) whether `addBlocks`/`addBlockedBy` dependency edges are enforced at runtime (reject create/update that would form a cycle or depend on an unknown id) or purely advisory metadata, and (c) whether the state-machine guards live in the `execute` handler or in a separate pure reducer that is also used by the reconstruction loop.

4. **State persistence — reconstruct-from-toolResults vs `pi.appendEntry` + compaction hazard.** Trace how `reconstructTodoState(ctx)` at `/Users/sguslystyi/rpiv-pi/extensions/rpiv-core/index.ts:426-439` walks `ctx.sessionManager.getBranch()` (see `ReadonlySessionManager` at `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/dist/core/session-manager.d.ts:136` and `SessionManager.getBranch()` at `session-manager.d.ts:244`) filtering `entry.type === "message" && msg.role === "toolResult" && msg.toolName === "todo"`, then compare against the alternative `pi.appendEntry<T>(customType, data)` path at `types.d.ts:761` plus `CustomEntry<T>` at `session-manager.d.ts:65-69` — the mechanism rpiv-core already uses at line 217 for the `active_agent` root entry. Critically: `rpiv-core/index.ts:271-273` only clears guidance markers on `session_compact`, so when compaction prunes old toolResult entries (see `SessionBeforeCompactEvent.branchEntries: SessionEntry[]` and `preparation: CompactionPreparation` at `types.d.ts:344-350`) an `in_progress` task will silently vanish. Decide whether the upgrade adds a `session_before_compact` handler that snapshots state into a `CustomEntry` or moves persistence entirely to `appendEntry`, and whether the two `session_start` handlers (the scaffolding one at `rpiv-core/index.ts:206` and the replay one at `rpiv-core/index.ts:441`) need to be merged or re-ordered so the replay sees a fully-initialised context.

5. **Rendering — `/todos` overlay + tool-row `renderCall`/`renderResult` growing to show status, activeForm, and dependencies.** Trace how the Pi example's `TodoListComponent` at `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/examples/extensions/todo.ts:40-103` implements `render(width): string[]`, `handleInput(data)`, and `invalidate()` as a Pi `Component`, and is mounted via `ctx.ui.custom<void>((_tui, theme, _kb, done) => new TodoListComponent(...))` at `todo.ts:292-294` (see `ExtensionUIContext.custom` at `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/dist/core/extensions/types.d.ts:93-103`), vs rpiv-core's current `/todos` command at `/Users/sguslystyi/rpiv-pi/extensions/rpiv-core/index.ts:526-546` which dumps a toast via `ctx.ui.notify`. Also trace the per-action `renderResult` branches at `todo.ts:228-280` and the simpler themed `renderCall` at `todo.ts:221-226` against the only `renderCall`/`renderResult` pattern actually in rpiv-pi — `web_search` and `web_fetch` at `/Users/sguslystyi/rpiv-pi/extensions/web-tools/index.ts:247-272` and `407-441` using `Text` from `@mariozechner/pi-tui` and `theme.fg("toolTitle"|"accent"|"success"|"error", …)`. Design how a richer Task list overlay would group by status (`pending`/`in_progress`/`completed`), show the `activeForm` spinner inline next to in_progress items, expose blockedBy edges without exceeding the per-tool-row height Pi reserves, and handle keyboard focus (escape to close, up/down for navigation) via `matchesKey` from `@mariozechner/pi-tui`.

6. **Permissions & existing-user migration hazard — `/Users/sguslystyi/rpiv-pi/extensions/rpiv-core/templates/pi-permissions.jsonc:26` hard-codes `"todo": "allow"`, and `seedPermissionsFile()` at `/Users/sguslystyi/rpiv-pi/extensions/rpiv-core/index.ts:151-167` only writes `~/.pi/agent/pi-permissions.jsonc` when it is ABSENT (the check at line 152).** Trace what happens for an existing rpiv-pi user on the first upgraded session: the `session_start` handler at `rpiv-core/index.ts:246-251` silently no-ops, the already-seeded `/Users/sguslystyi/.pi/agent/pi-permissions.jsonc:26` still says `"todo": "allow"` and knows nothing about a new `task`/`TaskCreate`/`TaskUpdate`/`TaskList`/`TaskGet` entry, and the first LLM call to the upgraded tool falls through to `defaultPolicy.tools: "ask"` at template line 12 — which is an interactive prompt that blocks any non-UI run and interrupts every interactive run. Decide (a) whether the new tool keeps the name `"todo"` for seamless backward compatibility, (b) whether `seedPermissionsFile()` grows a migration path that upserts new allow-list entries into an existing file (parse + mutate the JSONC, with a comment header), (c) whether `/rpiv-setup` at `rpiv-core/index.ts:572-655` gets a companion `/rpiv-migrate-permissions` action, or (d) whether the session_start handler notifies the user when it detects an out-of-date permissions file so they can re-run setup manually.
