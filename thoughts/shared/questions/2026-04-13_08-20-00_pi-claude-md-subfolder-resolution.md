# Research Questions: Pi CLAUDE.md subfolder resolution + extension adoption for annotate-inline

## Discovery Summary

Pi's SDK (`@mariozechner/pi-coding-agent`) already has an `AGENTS.md`/`CLAUDE.md` loader at `dist/core/resource-loader.js:30-76` — `loadContextFileFromDir(dir)` tries `["AGENTS.md","CLAUDE.md"]` (first wins per dir), `loadProjectContextFiles({cwd, agentDir})` walks from `agentDir` (global) + ancestor chain **upward from cwd to filesystem root**, dedupes by absolute path, and feeds the result into `buildSystemPrompt` at `dist/core/system-prompt.js:21-27, 100-106` where each file is rendered as `## <path>\n\n<content>` under a `# Project Context` heading. Crucially, this walk does **not descend** into subfolders of cwd — so nested `CLAUDE.md` files at `src/core/CLAUDE.md`, `src/services/CLAUDE.md` (exactly what the `annotate-inline` skill produces per `skills/annotate-inline/SKILL.md:270-282, 299`) are **never loaded** by Pi. The parallel `.rpiv/guidance/` subsystem in `extensions/rpiv-core/guidance.ts:22-107` already solves a nearly identical problem in the opposite direction: on each `tool_call` for `read|edit|write`, `resolveGuidance(filePath, projectDir)` walks **from the touched file's directory up to project root**, collects `<cwd>/.rpiv/guidance/<subPath>/architecture.md` at each depth, dedupes via a module-level `Set<string>`, and injects via `pi.sendMessage({ customType:"rpiv-guidance", display:false })`, wired to `pi.on("tool_call")` in `index.ts:135-137` with state-reset hooks on `session_start`/`session_compact`/`session_shutdown`. The SDK exposes two extension seams that could host a CLAUDE.md subfolder resolver without `pi.sendMessage`: `DefaultResourceLoaderOptions.agentsFilesOverride` (`resource-loader.d.ts:87-95`, composable transformation of the context-files array before system-prompt build) and the `beforeAgentStart` lifecycle hook returning a modified `systemPrompt` (`dist/core/extensions/runner.js:581-626`, `agent-session.js:767-772`). The thoughts/ chain already documents the CC→Pi migration (`thoughts/shared/{questions,research,designs,plans}/2026-04-10_*complete-pi-migration.md` + `thoughts/MIGRATION.md`) but contains no dedicated analysis of Pi's CLAUDE.md auto-loading behavior or nested-file support.

## Questions

1. Exactly how does Pi's resource loader walk ancestors to assemble `agentsFiles`, and where is the boundary that stops it from descending into subfolders? Trace `loadProjectContextFiles({cwd, agentDir})` at `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/dist/core/resource-loader.js:48-76` — how it seeds with `loadContextFileFromDir(resolvedAgentDir)` (line 53), then walks `currentDir = resolvedCwd` upward via `resolve(currentDir, "..")` until `parentDir === currentDir` or `currentDir === root`, `unshift`-ing matches into `ancestorContextFiles` and deduping via `seenPaths`. Explain why the walk is strictly upward (no `readdirSync` recursion), what ordering the caller sees (global first, then shallowest ancestor first, cwd last), and confirm by reading the full file that there is no separate path that finds CLAUDE.md files **below** cwd. This establishes the precise feature gap that the new extension must fill.

2. How does `buildSystemPrompt` consume the `agentsFiles` array, and what does that imply about cost, freshness, and injection format for a subfolder-aware loader? Trace `/usr/local/lib/node_modules/@mariozechner/pi-coding-agent/dist/core/system-prompt.js:7-106` — both the `customPrompt` branch (lines 15-27) and default branch (lines 99-106) render each `{path, content}` as `## <filePath>\n\n<content>\n\n` under a `# Project Context` heading. Cross-reference `agent-session.js:643-647` (`_rebuildSystemPrompt` calling `resourceLoader.getAgentsFiles().agentsFiles`) and the rebuild call sites at `agent-session.js:558, 625, 1629`. Answer: if an extension adds *every* subfolder CLAUDE.md here, is the full text sent with every LLM turn? Does `/reload` reread them (per `CHANGELOG.md:1303`)? This shapes whether nested injection should be eager (system-prompt-level) or lazy (per `tool_call` message, like `guidance.ts`).

3. What composable seam does the SDK offer to override the context-file list without forking Pi, and how does it compose with `beforeAgentStart`? Read `DefaultResourceLoaderOptions.agentsFilesOverride` at `resource-loader.d.ts:87-95` — `(base: {agentsFiles}) => {agentsFiles}` — and locate its invocation in `DefaultResourceLoader.reload()` (referenced at `resource-loader.js:320-322`). Separately, trace `emitBeforeAgentStart` in `dist/core/extensions/runner.js:581-626` returning `{systemPrompt?, messages?, display?}` and its application at `agent-session.js:767-772`. Document which seam an **extension** (as opposed to an embedding SDK caller) can actually reach — `agentsFilesOverride` is an option on the loader constructor, not on `ExtensionAPI`; `beforeAgentStart` is what extensions can use. This determines the *only* legitimate injection mechanisms for the new extension.

4. What is the precise algorithm and state model of the existing guidance extension that a CLAUDE.md-nested loader could reuse or parallel? Trace `resolveGuidance(filePath, projectDir)` in `extensions/rpiv-core/guidance.ts:22-51` — `dirname(filePath)` → `relative(projectDir, fileDir)` guard against traversal (lines 27-29) → loop `for (let depth = 0; depth <= parts.length; depth++)` joining `.rpiv/guidance/<subPath>/architecture.md` (lines 34-48), returning root-first. Then trace `handleToolCallGuidance` at lines 72-107 — the `["read","edit","write"]` gate, the `injectedGuidance: Set<string>` at line 58, `clearInjectionState()` wired in `index.ts:42,108,115`, and the `pi.sendMessage({customType:"rpiv-guidance", display:false})` call at lines 102-106. Identify the exact generalization point: the filename pattern and the injection label/`customType` are the only differences between a "guidance" walker and a "CLAUDE.md" walker.

5. Which architectural shape should the new CLAUDE.md-subfolder extension take — a second `tool_call` handler using `pi.sendMessage`, an `agentsFilesOverride` composition, or a `beforeAgentStart`-returned `systemPrompt` — and what are the trade-offs visible from the code? Compare (a) the reactive per-file path in `guidance.ts` + `pi.on("tool_call", ...)` at `index.ts:135-137` injecting `display:false` messages, against (b) eager pre-walking all CLAUDE.md files beneath `ctx.cwd` once on `session_start` and returning them as `{message:{customType,content,display:false}}` from `before_agent_start` (pattern already used for git context at `index.ts:140-157`), against (c) forking Pi to expose `agentsFilesOverride` on `ExtensionAPI`. Read `guidance.ts`, the `before_agent_start` git-context implementation, and the `resource-loader.d.ts` override interface to show what each path costs in token budget, staleness risk, and session-replay behavior (recall `injectedGuidance.clear()` on compact — does the same need to happen for CLAUDE.md?).

6. How should the new extension be refactored to share code with `guidance.ts` without coupling their filename conventions or `customTypes`? Propose the exact module shape given `guidance.ts:22-51` — e.g. extracting `walkAncestorsCollecting(filePath, projectDir, resolveFn: (subPath) => string)` into a shared `resolve-ancestors.ts`, with `guidance.ts` binding it to `join(".rpiv/guidance", subPath, "architecture.md")` and a new `claude-md.ts` binding it to the direct `<subPath>/CLAUDE.md` path. Additionally, propose how the dedup `Set<string>` should be scoped — shared module singleton, per-subsystem singleton, or keyed by `(subsystem, relativePath)` — and which of `session_start`, `session_compact`, `session_shutdown` in `index.ts:41-118` need to reset the new state. Name the exact exports (`resolveClaudeMd`, `handleToolCallClaudeMd`, `clearClaudeMdInjectionState`) to mirror `guidance.ts`'s API shape and preserve the boundary rule at `extensions/rpiv-core/CLAUDE.md:74-76` ("NO ExtensionAPI in utility modules").

7. What does `annotate-inline` actually produce, and does Pi's existing ancestor-only loader at least cover the root `CLAUDE.md` (so the new extension only needs to handle subfolders)? Trace `skills/annotate-inline/SKILL.md:41-73, 159-176, 200-217, 270-282` — the skill detects architecture, proposes targets including a root `CLAUDE.md` and subfolder files like `src/core/CLAUDE.md`, drafts subfolder files first (max 100 lines) then root (compact), and `Write`s them all at once at Pass 3. Cross-reference Pi's loader behavior: `resource-loader.js:62` calling `loadContextFileFromDir(currentDir)` at each ancestor including `cwd` itself — so `<cwd>/CLAUDE.md` **is** already picked up when Pi is launched from that cwd. Confirm whether subfolders (`extensions/rpiv-core/CLAUDE.md`, `scripts/CLAUDE.md`, `skills/CLAUDE.md`) are ignored today despite existing, which pinpoints that the new extension's responsibility is *only* subfolders-below-cwd, never the root or ancestors (which Pi already does and would duplicate).

8. What dedup key and conflict-handling rules must the new extension adopt to avoid double-injecting CLAUDE.md content that Pi's resource loader already injected into the system prompt? A subfolder `CLAUDE.md` discovered by the new extension at, say, `src/core/CLAUDE.md`, would not collide with what Pi's `loadProjectContextFiles` already loaded (which only covers cwd+ancestors), so collisions only arise if the extension's walk reaches a file Pi also resolved — i.e. when `tool_call`'s `file_path` sits at or above cwd. Trace the boundary guard in `guidance.ts:27-29` (`relativeDir.startsWith("..") || isAbsolute(relativeDir)` → return `[]`) to show how out-of-project paths are already excluded, and propose how the new extension should additionally skip the cwd-root `CLAUDE.md` (to avoid duplicating Pi's own loader). Also clarify the `AGENTS.md` vs `CLAUDE.md` precedence mirrored from `resource-loader.js:30-46` (AGENTS.md wins first per dir) — should the subfolder walker adopt the same ordering for consistency, and what `customType` distinguishes these messages from guidance in any future `pi.registerMessageRenderer` handler?
